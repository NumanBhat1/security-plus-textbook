# Volume 4 — Domain 4.0: Security Operations

# Objective 4.1 — Apply common security techniques to computing resources

> **Official CompTIA Objective Statement (SY0-701, v5.0):**
> *Given a scenario, apply common security techniques to computing resources.*
>
> Official sub-bullets:
> - **Secure baselines** — Establish; Deploy; Maintain
> - **Hardening targets** — Mobile devices; Workstations; Switches; Routers; Cloud infrastructure; Servers; ICS/SCADA; Embedded systems; RTOS; IoT devices
> - **Wireless devices** — Installation considerations (Site surveys; Heat maps)
> - **Mobile solutions** — Mobile device management (MDM); Deployment models (BYOD, COPE, CYOD); Connection methods (Cellular, Wi-Fi, Bluetooth)
> - **Wireless security settings** — Wi-Fi Protected Access 3 (WPA3); AAA/Remote Authentication Dial-In User Service (RADIUS); Cryptographic protocols; Authentication protocols
> - **Application security** — Input validation; Secure cookies; Static code analysis; Code signing
> - **Sandboxing**
> - **Monitoring**

---

## 3. Learning Outcomes

After this objective you will be able to:

1. Establish, deploy, and maintain **secure baselines**.
2. Apply **hardening** to all the official target types (workstations, servers, mobile, network gear, ICS/IoT).
3. Plan wireless installs (**site surveys, heat maps**) and configure **WPA3 / RADIUS / 802.1X**.
4. Compare **mobile deployment models** (BYOD/COPE/CYOD) and **MDM**.
5. Apply **application security** (input validation, secure cookies, static analysis, code signing) and **sandboxing**.

---

## 4. Concept Overview

This objective is the practical "lock it down" toolkit: how to take any computing resource and make it secure. The spine is the **secure baseline** (a known-good secure configuration you **establish → deploy → maintain**) plus **hardening** (reducing each device's attack surface). It then layers on wireless/mobile security, application security, and sandboxing.

Exam-wise, expect "given this device/scenario, which technique?" Know the **WPA3/RADIUS** wireless settings, the **BYOD vs COPE vs CYOD** distinction, and the **application security** terms (input validation, code signing, static vs dynamic analysis).

## 6. Deep Technical Explanation

### Secure baselines (establish → deploy → maintain)

A **baseline** is your defined secure configuration (e.g., a CIS Benchmark for Windows Server).

- **Establish** — define the secure config (ports off, services minimized, settings hardened).
- **Deploy** — push it to systems (via GPO, MDM, config management/IaC).
- **Maintain** — keep systems on the baseline; detect/remediate drift (configuration enforcement — 2.5).

### Hardening targets

Hardening = applying the baseline + reducing attack surface, tailored per device:

| Target | Key hardening |
|--------|---------------|
| **Workstations** | Patch, EDR, disk encryption, least privilege, disable unused services |
| **Servers** | Minimal roles, patch, host firewall, baseline configs |
| **Mobile devices** | MDM, encryption, screen lock, remote wipe |
| **Switches/Routers** | Change default creds, disable unused ports, secure mgmt, 802.1X |
| **Cloud infrastructure** | Least-privilege IAM, secure config, CSPM |
| **ICS/SCADA, RTOS, embedded, IoT** | **Often can't patch → segment/isolate, change defaults, monitor** |

> Recurring theme: ICS/SCADA/RTOS/embedded/IoT are **hard to patch** → harden by **segmentation/isolation + changing defaults + monitoring** (ties to 3.1).

### Wireless: installation + security

- **Site survey** — assess the physical space to plan AP placement, channels, and coverage.
- **Heat map** — visualizes signal strength to find dead zones / overlap / leakage outside the building.
- **Security settings:**
  - **WPA3** — strongest Wi-Fi security (uses **SAE** — replaces the WPA2 PSK handshake, resists offline cracking).
  - **AAA / RADIUS** — enterprise auth: users authenticate to a central **RADIUS** server (via **802.1X**), not a shared password.
  - **Cryptographic protocols** — AES/CCMP (WPA2) → SAE/AES (WPA3).
  - **Authentication protocols** — EAP variants (EAP-TLS, PEAP) for enterprise.

> **Personal (PSK) vs Enterprise:** PSK = one shared passphrase; **Enterprise (802.1X/RADIUS)** = each user authenticates individually (better for orgs).

### Mobile solutions

- **MDM** — centrally enforce policy on mobile devices (encryption, passcode, remote wipe, app control).
- **Deployment models:**

| Model | Who owns it | Note |
|-------|-------------|------|
| **BYOD** | **Employee** | Cheapest; hardest to secure/control |
| **COPE** | **Company** (some personal use allowed) | Company control + user convenience |
| **CYOD** | **Company** (employee chooses from approved list) | Company-owned, user picks model |

> **BYOD vs COPE vs CYOD trap:** BYOD = **employee-owned**; COPE = **company-owned, personal use OK**; CYOD = **company-owned, employee chose the model**.

- **Connection methods** — cellular, Wi-Fi, Bluetooth (each a vector — secure/limit as needed).

### Application security

| Technique | What it does |
|-----------|--------------|
| **Input validation** | Sanitize/validate input → stops injection (SQLi/XSS) |
| **Secure cookies** | Set **Secure** (HTTPS-only) + **HttpOnly** (no JS access) flags |
| **Static code analysis (SAST)** | Analyze **source code without running** it |
| **Dynamic analysis (DAST)** | Test the app **while running** |
| **Code signing** | Sign code with a cert → proves authenticity + integrity (detects tampering) |

> **Static vs dynamic:** static = **read the code** (not running); dynamic = **run and test** it. **Input validation** is the #1 defense against injection.

### Sandboxing

Running untrusted code/apps in an **isolated environment** so it can't harm the host — used for malware analysis, testing, and untrusted attachments/links.

### Monitoring

Continuously observe these resources (logs, EDR, performance) to detect issues — leads into 4.4.

## 7. Why It Exists

Every device ships insecure by default (open services, default creds). This objective gives you the repeatable techniques — baseline, harden, secure wireless/mobile/apps, sandbox — to bring any resource to a defensible state. Baselines make security **consistent and scalable**; hardening **shrinks the attack surface** attackers rely on.

---

## 8. How It Works (enterprise)

Orgs define **baselines** (CIS Benchmarks), deploy them via GPO/MDM/IaC, and enforce them. Devices are **hardened** by type; ICS/IoT that can't be patched are **segmented**. Wireless is planned with **site surveys/heat maps** and secured with **WPA3 + RADIUS/802.1X**. Mobile is governed by **MDM** and a **deployment model**. Apps are secured in the SDLC (**input validation, SAST/DAST, code signing**), and risky code runs in **sandboxes**. Everything is **monitored**.

---

## 9. Advantages

- **Consistent, scalable security** (baselines).
- **Smaller attack surface** (hardening).
- **Strong wireless/mobile control** (WPA3/RADIUS/MDM).
- **Fewer app vulnerabilities** (validation, SAST/DAST, signing).
- **Contained risk** (sandboxing).

---

## 10. Disadvantages / Challenges

- **Maintenance effort** — baselines drift; hardening must be sustained.
- **Unpatchable devices** (ICS/IoT) limit hardening to compensating controls.
- **BYOD** is hard to secure (personal devices).
- **App security** requires SDLC integration and developer buy-in.
- **Over-hardening** can break functionality.

---

## 11. Real-World Examples

- **Baseline:** a CIS-hardened Windows image deployed to all workstations.
- **Heat map:** a survey reveals Wi-Fi leaking into the parking lot → adjust AP power.
- **WPA3 + RADIUS:** corporate Wi-Fi where each user authenticates via 802.1X.
- **MDM + COPE:** company phones with enforced encryption + remote wipe.
- **Code signing:** an app rejected because its signature is invalid (tampered).
- **Sandbox:** an email attachment detonated in a sandbox before delivery.

---

## 12. Enterprise Usage (SOC relevance)

- **Baseline drift** and **missing hardening** show up as findings you report.
- **Rogue/weak wireless** and **unmanaged mobile** devices are alerts you investigate.
- **Sandbox detonation** of suspicious files is part of malware triage.
- **Code-signing failures** can indicate tampering/supply-chain issues.
- **Monitoring** these resources feeds your SIEM.

---

## 13. Security Implications

- **No baseline** = inconsistent, exploitable configs.
- **Default creds / open ports** = trivial compromise.
- **Weak Wi-Fi (WEP/WPA, PSK)** = easy wireless attacks.
- **Unmanaged BYOD** = data leakage, malware entry.
- **No input validation** = injection (SQLi/XSS).
- **Unsigned code** = tampering/malware risk.

---

## 14. Attacker / Adversary Perspective

- Attackers exploit **default credentials, open ports, unhardened** devices first.
- They target **weak wireless** (WEP/WPA, evil twins) and **leaking signal**.
- They abuse **unmanaged mobile** and **BYOD** as entry points.
- They exploit **poor input validation** (injection) and **unsigned/tampered code**.
- **Unpatchable ICS/IoT** are prime targets — hence segmentation.

---

## 15. Defender Perspective

- **Establish, deploy, and maintain baselines;** enforce config.
- **Harden every device type;** segment what you can't patch.
- **Plan wireless** (survey/heat map); use **WPA3 + RADIUS/802.1X**.
- **Manage mobile with MDM;** pick the right deployment model.
- **Build app security in** (validation, SAST/DAST, code signing).
- **Sandbox** untrusted code; **monitor** everything.

---

## 16. Detection / Tooling

- **Configuration/compliance scanners** (SCAP, CIS-CAT) for baseline drift.
- **MDM consoles** for mobile posture.
- **Wireless IDS** for rogue APs/evil twins.
- **SAST/DAST tools** in CI/CD.
- **Sandboxes** for file/URL detonation.

---

## 17. Mitigation / Hardening Techniques

- Baselines + configuration enforcement; patching.
- Change default credentials; disable unused ports/services/protocols.
- WPA3/RADIUS/802.1X; MDM + encryption + remote wipe.
- Input validation, secure cookies, code signing, SAST/DAST.
- Segmentation/isolation for ICS/IoT; sandboxing for untrusted code.

---

## 18. Best Practices

1. **Establish baselines (CIS), deploy widely, maintain/enforce them.**
2. **Harden by device type;** segment unpatchable ICS/IoT.
3. **Plan wireless** with surveys/heat maps; use **WPA3 + 802.1X/RADIUS**.
4. **Use MDM** and the right deployment model (COPE/CYOD over raw BYOD where possible).
5. **Bake app security into the SDLC** (validation, SAST/DAST, signing).
6. **Sandbox untrusted code; monitor continuously.**

## 20. Diagrams

**Baseline lifecycle:**

```
   ESTABLISH (define secure config) → DEPLOY (GPO/MDM/IaC) → MAINTAIN (enforce, fix drift) → (repeat)
```

**Enterprise Wi-Fi auth (802.1X/RADIUS):**

```
   Device (supplicant) → AP/switch (authenticator) → RADIUS server (auth)
   each user authenticates individually (not a shared PSK)
```

**Static vs dynamic:**

```
   STATIC (SAST): read the source code — app NOT running
   DYNAMIC (DAST): run the app and test it — app RUNNING
```

## 29. Practice Questions

**Q1.** A known-good secure configuration applied consistently across all systems is called a:
A. Heat map  B. Secure baseline  C. Sandbox  D. Honeypot

**Q2.** Which Wi-Fi security standard is the strongest and uses SAE?
A. WEP  B. WPA  C. WPA2  D. WPA3

**Q3.** A company wants each user to authenticate individually to the corporate Wi-Fi via a central server. They should use:
A. WPA2-PSK  B. 802.1X with RADIUS  C. WEP  D. A shared passphrase

**Q4.** Employees use their own personally owned smartphones for work. This deployment model is:
A. COPE  B. CYOD  C. BYOD  D. MDM

**Q5.** Reviewing application source code for vulnerabilities WITHOUT executing the program is:
A. Dynamic analysis  B. Static analysis  C. Fuzzing  D. Sandboxing

**Q6.** Signing software with a certificate to prove it is authentic and unmodified is:
A. Input validation  B. Code signing  C. Hashing only  D. Tokenization

**Q7.** Running a suspicious email attachment in an isolated environment to observe its behavior is:
A. Hardening  B. Sandboxing  C. Baseline deployment  D. A heat map

**Q8.** A visual representation of wireless signal strength across a building is a:
A. Site survey  B. Heat map  C. Baseline  D. Topology diagram

**Q9.** Which is the BEST way to secure ICS/SCADA systems that cannot be patched?
A. Ignore them  B. Segment/isolate them and monitor closely  C. Connect them directly to the internet  D. Use WEP

**Q10.** The primary application defense against SQL injection and XSS is:
A. Code signing  B. Input validation  C. Secure cookies  D. Heat mapping

### Answers
- **Q1 — B** (Secure baseline). **Q2 — D** (WPA3/SAE). **Q3 — B** (802.1X + RADIUS). **Q4 — C** (BYOD). **Q5 — B** (Static). **Q6 — B** (Code signing). **Q7 — B** (Sandboxing). **Q8 — B** (Heat map). **Q9 — B** (Segment/isolate + monitor). **Q10 — B** (Input validation).
