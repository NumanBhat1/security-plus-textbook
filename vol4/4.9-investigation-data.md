# Volume 4 — Domain 4.0: Security Operations

# Objective 4.9 — Use data sources to support an investigation

> **Official CompTIA Objective Statement (SY0-701, v5.0):**
> *Given a scenario, use data sources to support an investigation.*
>
> Official sub-bullets:
> - **Log data** — Firewall logs; Application logs; Endpoint logs; OS-specific security logs; IPS/IDS logs; Network logs; Metadata
> - **Data sources** — Vulnerability scans; Automated reports; Dashboards; Packet captures

---

## 3. Learning Outcomes

After this objective you will be able to:

1. Identify which **log source** answers which investigative question.
2. Use **firewall, application, endpoint, OS security, IPS/IDS, and network logs** + **metadata**.
3. Use **vulnerability scans, automated reports, dashboards, and packet captures** as investigation data.
4. Match a scenario to the **right data source**.

---

## 4. Concept Overview

When investigating an incident, you need **evidence from the right source**. This objective is about knowing **which log or data source answers which question** — the everyday SOC skill of "where do I look?" CompTIA gives a scenario ("you need to see X") and asks which data source.

The key skill: map the question to the source (e.g., "what connected to what" → firewall/network logs; "what ran on the host" → endpoint logs; "the actual traffic contents" → packet capture).

---

## 5. Definitions

| Term | Definition |
|------|------------|
| **Firewall logs** | Records of allowed/denied connections (IP/port). |
| **Application logs** | Events from a specific application. |
| **Endpoint logs** | Activity on a host (processes, files). |
| **OS security logs** | OS-level security events (logins, privilege use). |
| **IPS/IDS logs** | Detected/blocked intrusion events. |
| **Network logs** | Traffic/connection records (routers, switches, flow). |
| **Metadata** | Data about data (timestamps, sender, file properties). |
| **Vulnerability scan** | Report of weaknesses found. |
| **Automated report** | Scheduled summary from a tool. |
| **Dashboard** | Visual, real-time view of metrics/alerts. |
| **Packet capture (PCAP)** | The **actual contents** of network traffic. |

---

## 6. Deep Technical Explanation

### Log sources — which answers which question

| Log | Answers… | Use for |
|-----|----------|---------|
| **Firewall logs** | What connections were **allowed/blocked** (IP/port) | Tracking connections, blocked attacks |
| **Application logs** | What happened **inside an app** (errors, transactions) | App-specific incidents |
| **Endpoint logs** | What ran **on the host** (processes, file changes) | Malware/host compromise |
| **OS security logs** | **Logins, privilege use, account changes** | Authentication/privilege investigations |
| **IPS/IDS logs** | **Detected/blocked intrusions** | Attack detection details |
| **Network logs** | **Traffic/connections** across the network | Lateral movement, exfil patterns |
| **Metadata** | **Context** (timestamps, sender, geolocation, file info) | Email headers, file origin, timelines |

> **Mapping the question to the log is the whole skill:**
> - "Who logged in / privilege escalation" → **OS security logs**.
> - "What connected to what (allowed/blocked)" → **firewall logs**.
> - "What process ran on the machine" → **endpoint logs**.
> - "Was this attack detected/blocked" → **IPS/IDS logs**.
> - "Email sender/timestamp/route" → **metadata** (email headers).

### Other data sources

| Source | What it gives |
|--------|---------------|
| **Vulnerability scans** | Weaknesses present (context for how they got in) |
| **Automated reports** | Scheduled summaries (trends, compliance) |
| **Dashboards** | Real-time visual overview (quick situational awareness) |
| **Packet captures (PCAP)** | The **actual traffic contents** (deepest network detail) |

> **Packet capture vs logs/NetFlow:** logs/NetFlow tell you **that** a connection happened (metadata); a **packet capture shows the actual content** of the traffic (the deepest detail — what was actually sent). "Need to see the actual data/payload transmitted" = **packet capture**.

> **Metadata** is heavily testable: it's "data about data" — email **headers** (sender, route, timestamps), file properties, photo geolocation. "Examine the email header to find the true sender" = metadata.

---

## 7. Why It Exists

Investigations live or die on **evidence**. Knowing which source holds the answer makes you fast and accurate; looking in the wrong place wastes time. This objective builds the analyst reflex: given a question, instantly know the data source.

---

## 8. How It Works (enterprise)

All sources feed (or are queried via) the **SIEM**, which correlates logs from firewalls, endpoints, OS, IDS/IPS, and network. Analysts pivot across sources during an investigation: a **dashboard** flags an anomaly → **firewall/network logs** show the connection → **endpoint logs** show the process → **packet capture** reveals the payload → **metadata** confirms the email origin. **Vulnerability scans** and **automated reports** add context.

---

## 9. Advantages

- **Right evidence, fast** (know where to look).
- **Correlation** (SIEM ties sources together).
- **Depth when needed** (packet capture).
- **Situational awareness** (dashboards).
- **Context** (vuln scans, metadata).

---

## 10. Disadvantages / Challenges

- **Volume** — huge log data; need correlation/filtering.
- **Packet capture** is storage-heavy and may be encrypted.
- **Missing logs** = blind spots (anti-forensics).
- **Metadata can be spoofed** (verify carefully).

---

## 11. Real-World Examples

- **Phishing investigation:** email **metadata/headers** reveal the true sender and route.
- **Malware on a host:** **endpoint logs** show the malicious process; **packet capture** shows its C2 traffic.
- **Account compromise:** **OS security logs** show the suspicious login + privilege use.
- **Blocked attack:** **firewall/IPS logs** show the denied connection.
- **Exfil:** **network logs/NetFlow** show the large outbound transfer; **PCAP** confirms the content.

---

## 12. Enterprise Usage (SOC relevance)

This is the **core analyst skill** — every investigation is "pull the right data." You'll:
- Pivot across **logs** in the SIEM.
- Read **email headers (metadata)** for phishing.
- Pull **packet captures** for deep analysis.
- Use **dashboards** for triage and **automated reports** for trends.
- Correlate **vuln scans** to explain the entry point.

---

## 13. Security Implications

- **Missing/insufficient logs** cripple investigations (log everything important; protect integrity).
- **No packet capture** = can't see payloads when needed.
- **Spoofable metadata** must be corroborated.
- **Short retention** = evidence gone (tie to 5.4/4.4 archiving).

---

## 14. Attacker / Adversary Perspective

- Attackers **delete/disable logs** (missing logs — 2.4) to blind investigators.
- They **encrypt traffic** to defeat packet inspection.
- They **spoof metadata** (forged email headers).
- They exploit **gaps in logging coverage**.

---

## 15. Defender Perspective

- **Log comprehensively** (firewall, endpoint, OS, IDS/IPS, network, app) → centralize in SIEM.
- **Protect log integrity** and retention.
- **Capture packets** where deep visibility is needed.
- **Know the source-to-question mapping** cold.
- **Corroborate metadata** (don't trust spoofable fields alone).

---

## 16. Tooling

- **SIEM** (log correlation), **EDR** (endpoint), **packet analyzers (Wireshark/tcpdump)**.
- **NetFlow** for traffic patterns.
- **Vulnerability scanners**, **dashboards**, **automated reporting**.

---

## 17. Best Practices

1. **Log all critical sources;** centralize and correlate (SIEM).
2. **Protect log integrity + retention** (don't let "missing logs" happen).
3. **Capture packets** for deep analysis when needed.
4. **Master the source→question mapping** (fast investigations).
5. **Corroborate metadata;** treat it as spoofable.
6. **Use dashboards for triage, reports for trends.**

## 19. Diagrams

   PACKET CAPTURE            → the actual bytes sent                   (WHAT was sent)

## 20. Exam Tips

- **Map the question to the source** — this is the whole objective.
- **OS security logs = logins/privilege; firewall logs = connections; endpoint logs = host processes; IPS/IDS logs = detected intrusions; network logs = traffic.**
- **Metadata = data about data** (email headers, timestamps, file/photo info) — "find the true email sender" = metadata.
- **Packet capture = the actual traffic CONTENTS** (deepest detail); logs/NetFlow only show that a connection happened.
- **Dashboard = real-time view; automated report = scheduled summary.**
- **Vulnerability scan = weaknesses present** (entry-point context).
- **Missing logs = a red flag** (anti-forensics).

## 21. Common CompTIA Question Styles

1. *"You need to see the ACTUAL contents of network traffic."* → Packet capture.
2. *"You need to determine the true sender and route of a suspicious email."* → Metadata (email headers).
3. *"You're investigating which user logged in and escalated privileges."* → OS security logs.
4. *"You need to see which connections were allowed or blocked by IP/port."* → Firewall logs.
5. *"You need to see which process executed on a compromised laptop."* → Endpoint logs.
6. *"You want a real-time visual overview of security alerts."* → Dashboard.
7. *"You need to know what intrusions were detected/blocked."* → IPS/IDS logs.
8. *"You want to know what vulnerabilities the entry host had."* → Vulnerability scan.

**Why distractors fail:** they offer a source that shows *that* something happened when the question needs the *content* (packet capture), or pick the wrong log layer. Anchor on *what the question actually asks to see*.

## 22. Interview Notes

- *"Logs vs packet capture?"* — Logs/NetFlow show that a connection occurred (metadata); a packet capture shows the actual content of the traffic.
- *"Where do you look for a login investigation?"* — OS-specific security logs.
- *"How do you find a phishing email's true origin?"* — The email metadata/headers.
- *"What's metadata?"* — Data about data: timestamps, senders, file/photo properties — useful but spoofable.
- *"Dashboard vs report?"* — Dashboard is real-time; reports are scheduled summaries.

---

## 23. Common Beginner Mistakes

1. **Choosing logs when the question needs packet capture** (content vs metadata).
2. **Picking the wrong log layer** (firewall vs endpoint vs OS).
3. **Forgetting metadata = email headers** for sender origin.
4. **Trusting metadata blindly** (it's spoofable).
5. **Confusing dashboard (real-time) and report (scheduled).**
6. **Ignoring "missing logs" as an indicator.**

## 28. Practice Questions

**Q1.** An investigator needs to see the ACTUAL contents of data transmitted over the network. Which source is BEST?
A. Firewall logs  B. NetFlow  C. Packet capture  D. Dashboard

**Q2.** To determine the true sender, timestamps, and routing of a suspicious email, you examine the:
A. Endpoint logs  B. Email metadata/headers  C. Firewall logs  D. Vulnerability scan

**Q3.** Which log source BEST shows which user account logged in and escalated privileges?
A. Firewall logs  B. OS-specific security logs  C. Network logs  D. Application logs

**Q4.** Which shows which connections were allowed or denied based on IP and port?
A. Endpoint logs  B. Firewall logs  C. Metadata  D. Dashboard

**Q5.** To see which malicious process executed on a compromised workstation, check the:
A. Firewall logs  B. Endpoint logs  C. IPS/IDS logs  D. Automated report

**Q6.** Which provides a real-time visual overview of security metrics and alerts?
A. Packet capture  B. Dashboard  C. Vulnerability scan  D. Chain of custody

**Q7.** Which BEST distinguishes packet captures from logs/NetFlow?
A. They are identical
B. Packet captures show the actual content; logs/NetFlow show that a connection occurred
C. Logs show content; packet captures show only metadata
D. Both only show metadata

**Q8.** Which source would show what intrusions were detected and blocked?
A. Application logs  B. IPS/IDS logs  C. Metadata  D. Dashboard

**Q9.** Which source helps explain what weaknesses an exploited host had before the breach?
A. Vulnerability scan  B. Packet capture  C. Dashboard  D. OS security logs

**Q10.** Why should metadata (e.g., email headers) be corroborated rather than fully trusted?
A. It's always encrypted  B. It can be spoofed/forged  C. It's never available  D. It's too large

### Answers
- **Q1 — C** (Packet capture). **Q2 — B** (Metadata/headers). **Q3 — B** (OS security logs). **Q4 — B** (Firewall logs). **Q5 — B** (Endpoint logs). **Q6 — B** (Dashboard). **Q7 — B** (content vs that-it-happened). **Q8 — B** (IPS/IDS logs). **Q9 — A** (Vulnerability scan). **Q10 — B** (spoofable).
