## 3. Learning Outcomes

After this objective you will be able to:

1. Define **audits** and **assessments** and explain **attestation**.
2. Distinguish **internal** (compliance, audit committee, self-assessment) from **external** (regulatory, examination, independent third-party) audits.

5. Answer CompTIA's audit and pen-test-type questions confidently.

---

## 4. Concept Overview

**Audits and assessments** are how an organization *verifies* that its security controls actually work — turning "we think we're secure" into evidence. An **audit** is a formal, structured examination against a standard/requirement (often producing an **attestation**). An **assessment** is a broader evaluation of security posture (including **penetration testing**, which actively tests defenses).

Two big buckets for the exam:

1. **Audits** — who performs them: **internal** (your own people/committee) vs. **external** (regulators or independent third parties). External audits carry more weight because of independence.
2. **Penetration testing** — the *types* (physical, offensive, defensive, integrated) and especially the **knowledge environments** (known/partially known/unknown = white/gray/black box) plus **recon** (passive vs. active). The environment terms are guaranteed exam points.

## 5. Definitions

| Term | Definition |
|------|------------|
| **Audit** | A formal, structured examination of controls against a standard/requirement. |
| **Assessment** | An evaluation of security posture (broader than an audit). |
| **Attestation** | A formal declaration/confirmation that controls are in place and effective. |
| **Internal audit** | An audit conducted by the organization's own staff/audit function. |
| **Compliance (internal)** | Internal checking that the org meets its requirements. |
| **Audit committee** | A governance body (often board-level) overseeing audit activities. |
| **Self-assessment** | An organization evaluating its own controls. |
| **External audit** | An audit by an outside, independent party. |
| **Regulatory audit** | An audit/exam by a government regulator. |
| **Examination** | A formal regulatory review (common in finance). |
| **Independent third-party audit** | An audit by an external, unbiased firm (e.g., for SOC 2/ISO). |
| **Penetration testing** | An authorized simulated attack to find and exploit weaknesses. |
| **Physical pen test** | Testing physical security (locks, badges, tailgating). |
| **Offensive pen test** | Simulating attacker actions (red team). |
| **Defensive pen test** | Testing/strengthening detection and response (blue team). |
| **Integrated pen test** | Combining offense and defense (purple team). |
| **Known environment** | Tester has full knowledge of the target (**white-box**). |
| **Partially known environment** | Tester has limited knowledge (**gray-box**). |
| **Unknown environment** | Tester has no prior knowledge (**black-box**). |
| **Reconnaissance** | Gathering information about a target. |
| **Passive reconnaissance** | Gathering info **without** directly interacting with the target (OSINT). |
| **Active reconnaissance** | Gathering info by **directly** interacting (scanning, probing). |

## 6. Deep Technical Explanation

### Attestation

An **attestation** is a formal statement confirming that controls exist and work — typically the *output* of an audit. A SOC 2 report includes an attestation by an auditor. Internally, a control owner might attest that a control is operating. Attestation = "we formally declare this is true," often signed and used as assurance to others.

### Internal audits

Performed by the organization's own people/functions:

| Type | What it is |
|------|-----------|
| **Compliance** | Internal verification the org meets its regulatory/policy requirements. |
| **Audit committee** | A board-level governance group overseeing audits and reporting (independence within the org). |
| **Self-assessment** | The org evaluating its own controls (honest but inherently less independent). |

Internal audits catch issues early and are cheaper/more frequent, but lack outside independence.

### External audits

Performed by outside parties — more credible because **independent**:

| Type | What it is |
|------|-----------|
| **Regulatory** | Conducted by a government regulator to enforce law. |
| **Examination** | A formal regulator review (common in banking/finance). |
| **Assessment** | A broader external evaluation of posture. |
| **Independent third-party audit** | An unbiased external firm (e.g., for SOC 2 Type II, ISO 27001). |

> **Internal vs. external (exam point):** internal = your own staff (cheaper, frequent, less independent); external = outside party (independent, more credible, required for many certifications/regulations).

### Penetration testing — types

| Type | Focus | Team color |
|------|-------|-----------|
| **Physical** | Physical controls — doors, locks, badges, tailgating, guards | — |
| **Offensive** | Simulating real attacks to break in | **Red team** |
| **Defensive** | Detecting and responding to attacks | **Blue team** |
| **Integrated** | Offense + defense working together to improve both | **Purple team** |

### Penetration testing — environments (knowledge level) — the high-yield terms

CompTIA SY0-701 uses *neutral* environment names; the classic box-color names are the same idea:

| Official term | Classic name | Tester knowledge |
|---------------|--------------|------------------|
| **Known environment** | **White-box** | **Full** knowledge (architecture, source, credentials) |
| **Partially known environment** | **Gray-box** | **Some/limited** knowledge |
| **Unknown environment** | **Black-box** | **No** prior knowledge (most realistic of an external attacker) |

> **Memorize the mapping: Known = White-box (full), Partially known = Gray-box (some), Unknown = Black-box (none).** More knowledge = faster/deeper test; less knowledge = more realistic external-attacker simulation. This trio is a near-guaranteed exam item.

### Reconnaissance: passive vs. active

The recon phase of a pen test (and of real attacks — ties to 2.4):

| Type | Definition | Examples | Detectable? |
|------|-----------|----------|-------------|
| **Passive** | Gather info **without touching** the target | OSINT: WHOIS, Google, social media, public records, DNS records | Hard to detect (no contact) |
| **Active** | Gather info by **directly interacting** | Port scanning (nmap), banner grabbing, probing | More detectable (touches target) |

> **Passive = no direct contact (stealthy, OSINT); Active = direct interaction (scanning, more detectable).** This distinction is heavily tested.

### Rules of engagement

All pen testing requires **authorization** and agreed **rules of engagement** (scope, allowed techniques, timing, contacts) — same concept introduced in 5.3. Testing without authorization is a crime.

## 7. Why It Exists

Audits and assessments exist because **controls can fail silently** — a firewall rule drifts, a process is skipped, a "secure" config regresses. You can't trust controls you don't verify. Audits provide independent assurance (to leadership, regulators, customers); pen tests provide *adversarial* proof that defenses actually stop attacks. Together they convert assumptions into evidence and drive remediation.

---

## 8. How It Works (enterprise)

Internal audit and self-assessments run continuously/periodically against policies and compliance requirements, reporting to the **audit committee** and leadership. **External/independent** audits (SOC 2, ISO 27001, regulatory exams) provide third-party assurance and **attestations** used with customers/regulators. **Penetration tests** — scoped via rules of engagement and an environment (known/partial/unknown) — actively probe defenses (offensive), validate detection (defensive), or both (integrated/purple), starting with **reconnaissance** (passive then active). Findings feed remediation and the risk register (5.2).

---

## 9. Advantages

- Independent verification that controls work (not just exist).
- **Adversarial proof** (pen tests find real exploitable paths).
- Required for certifications/regulations (external audits).
- Drives prioritized remediation; supports governance reporting.

---

## 10. Disadvantages

- **Point-in-time** — an audit/pen test is a snapshot; posture changes after.
- **Cost** — external audits and quality pen tests are expensive.
- **Scope-limited** — only what's in scope is tested.
- **Self-assessment bias** — internal results may be over-optimistic.
- **"Passed the audit ≠ secure"** (compliance ≠ security — 5.4).

---

## 11. Real-World Examples

- A SaaS company gets an **independent third-party** SOC 2 Type II audit and shares the **attestation** with customers.
- A bank undergoes a **regulatory examination**.
- An **internal audit** team self-assesses controls quarterly and reports to the **audit committee**.
- A **physical** pen test: a tester tailgates through a badge door to reach a server room.
- A **black-box (unknown)** external pen test simulates an outside attacker with no inside knowledge.
- **Passive recon:** harvesting employee emails from LinkedIn; **active recon:** running nmap against the perimeter.

---

## 12. Enterprise Usage (SOC relevance)

- **Defensive pen tests / purple teaming** directly involve you (the blue team) — they test whether your detections fire.
- **Red-team exercises** generate activity you should detect; debriefs improve your rules.
- **Audit findings** often become detection/hardening work items.
- Understanding **recon** helps you spot it: passive recon is nearly invisible, but **active recon (port scans)** is something your SIEM/IDS should alert on (ties to 4.4/2.4).
- You may collect evidence/logs to support **attestations** and audits.

---

## 13. Security Implications

- **Unaudited controls** drift and fail silently.
- **Out-of-scope systems** in a pen test remain untested blind spots.
- **Stale assurance** — a year-old audit may not reflect current risk.
- **Detecting active recon** early can stop an attack in its earliest phase.
- **Authorization matters** — unauthorized "testing" is illegal and dangerous.

---

## 14. Attacker / Adversary Perspective

- Real attackers follow the same **recon → exploit** flow pen testers simulate; **passive recon** lets them profile you undetected.
- They prefer the **unknown/black-box** reality — no inside help, so they probe and enumerate.
- They exploit gaps **between audits** (the snapshot problem).
- They target **physical** weaknesses (tailgating) that purely technical audits miss.

---

## 15. Defender Perspective

- Run **regular internal + periodic external** audits; don't rely on a single snapshot.
- Commission **pen tests** (offensive + defensive/purple) with clear scope/RoE.
- **Detect active recon** (scan alerts) and minimize **passive recon** exposure (reduce public info/OSINT footprint).
- Treat findings as **remediation work**, feed them to the risk register.
- Remember audits prove *compliance*; pen tests prove *resilience* — do both.

---

## 16. Detection Techniques

- **IDS/IPS + SIEM** detect **active recon** (port scans, enumeration).
- **Honeypots/honeytokens** catch attacker probing (deception — 1.2).
- **Log/EDR review** during defensive/purple exercises validates detection coverage.
- **Threat intel/OSINT monitoring** to see what passive recon could reveal about you.

---

## 17. Mitigation Techniques

- **Remediate audit/pen-test findings** (patch, harden, fix process).
- **Reduce OSINT footprint** to limit passive recon value.
- **Network segmentation + least privilege** to limit what a test/attacker reaches.
- **Continuous monitoring** to shrink the gap between point-in-time assessments.
- **Strong physical controls** (validated by physical pen tests).

---

## 18. Best Practices

1. **Combine internal and external** audits (frequency + independence).
2. **Pen test regularly** with clear **rules of engagement** and authorization.
3. **Pick the right environment** (known for depth, unknown for realism).
4. **Run purple-team/defensive** tests to validate detection, not just find holes.
5. **Treat findings as risk** — remediate and track.
6. **Don't equate passing an audit with being secure.**
7. **Detect active recon**; minimize passive-recon exposure.

## 22. Common CompTIA Question Styles

1. **Environment:** *"A tester is given full network diagrams and source code. Which environment?"* → Known (white-box).
2. *"A tester is given no prior information, simulating a true outsider."* → Unknown (black-box).
3. **Recon:** *"A tester gathers info from WHOIS and LinkedIn without touching the target."* → Passive reconnaissance.
4. *"A tester runs nmap against the target's IP range."* → Active reconnaissance.
5. **Internal vs external:** *"Which audit is conducted by an unbiased outside firm for a SOC 2 report?"* → Independent third-party (external).
6. **Attestation:** *"A formal statement confirming controls are implemented and effective is a(n)…"* → Attestation.
7. **Team:** *"A team focused on detecting and responding to the simulated attack is the…"* → Blue team (defensive).
8. **Physical:** *"A tester attempts to tailgate through a secure door. This is a…"* → Physical pen test.

**Why distractors fail:** they swap known/unknown, passive/active, internal/external, or red/blue/purple. Anchor on knowledge level, contact level, who performs it, and team color.

## 23. Interview Notes

- *"White-box vs. black-box pen test?"* — White = full knowledge (deep/efficient); black = no knowledge (realistic outsider); gray = in between.
- *"Passive vs. active recon?"* — Passive = OSINT, no contact; active = scanning/probing, detectable.
- *"Red vs. blue vs. purple?"* — Offense, defense, and the two working together to improve both.
- *"Internal vs. external audit?"* — Own staff vs. independent outside party; external is more credible/required for certs.
- *"What is attestation?"* — A formal declaration that controls are implemented/effective.

## 24. Common Beginner Mistakes

1. **Swapping known/unknown (white/black-box).**
2. **Confusing passive and active recon** (contact vs. no contact).
3. Calling an **internal self-assessment** independent.
4. Mixing **red (offense) and blue (defense)**; forgetting **purple = integrated**.
5. Thinking **passing an audit = secure**.
6. Forgetting **pen tests require authorization + RoE**.
7. Overlooking **physical** pen testing.

## 29. Practice Questions

**Q1.** A penetration tester is given complete network diagrams, source code, and credentials before testing. Which environment is this?
A. Unknown (black-box)  B. Partially known (gray-box)  C. Known (white-box)  D. Physical

**Q2.** A tester gathers target information only from WHOIS records, search engines, and social media, never contacting the target's systems. This is:
A. Active reconnaissance  B. Passive reconnaissance  C. A vulnerability scan  D. An examination

**Q3.** Which test most realistically simulates an external attacker with no inside information?
A. Known environment  B. Partially known environment  C. Unknown environment  D. Self-assessment

**Q4.** A formal declaration that an organization's controls are implemented and operating effectively is called a(n):
A. Examination  B. Attestation  C. Reconnaissance  D. Self-assessment

**Q5.** Which audit is performed by an unbiased outside firm and is typically required for SOC 2 certification?
A. Self-assessment  B. Internal compliance audit  C. Independent third-party audit  D. Audit committee review

**Q6.** A red team simulates real-world attacks while a blue team works to detect and respond, collaborating to improve both. This combined approach is a(n):
A. Physical pen test  B. Offensive pen test  C. Integrated (purple) pen test  D. Passive recon

**Q7.** A tester runs an nmap scan against the target's public IP range. This activity is:
A. Passive recon  B. Active recon  C. An attestation  D. A self-assessment

**Q8.** A tester attempts to tailgate an employee through a badge-controlled door to reach a data center. This is a:
A. Offensive pen test  B. Physical pen test  C. Gray-box test  D. Defensive pen test

**Q9.** Which is an example of an INTERNAL audit activity?
A. A government regulator's examination  B. An independent third-party SOC 2 audit  C. A self-assessment reviewed by the audit committee  D. A vendor's external assessment

**Q10.** In a gray-box (partially known) penetration test, the tester has:
A. No knowledge of the target  B. Full knowledge of the target  C. Limited/partial knowledge of the target  D. Only physical access

### Answer Key & Explanations

- **Q1 — C (Known / white-box).** Full knowledge provided.
- **Q2 — B (Passive recon).** No direct contact — OSINT only.
- **Q3 — C (Unknown / black-box).** No inside info = most realistic outsider.
- **Q4 — B (Attestation).** Formal declaration controls work.
- **Q5 — C (Independent third-party audit).** External, unbiased.
- **Q6 — C (Integrated / purple).** Red + blue together.
- **Q7 — B (Active recon).** Scanning = direct interaction.
- **Q8 — B (Physical pen test).** Tests physical controls.
- **Q9 — C.** Self-assessment + audit committee = internal.
- **Q10 — C (Limited/partial knowledge).** Definition of gray-box.
