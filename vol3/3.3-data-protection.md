# Objective 3.3 — Compare and contrast concepts and strategies to protect data

> **Official CompTIA Objective Statement (SY0-701, v5.0):**
> *Compare and contrast concepts and strategies to protect data.*
>
> Official sub-bullets:
> - **Data types** — Regulated; Trade secret; Intellectual property; Legal information; Financial information; Human- and non-human-readable
> - **Data classifications** — Sensitive; Confidential; Public; Restricted; Private; Critical
> - **General data considerations** — Data states (Data at rest; Data in transit; Data in use); Data sovereignty; Geolocation
> - **Methods to secure data** — Geographic restrictions; Encryption; Hashing; Masking; Tokenization; Obfuscation; Segmentation; Permission restrictions

---

## 3. Learning Outcomes

After this objective you will be able to:

1. Identify **data types** (regulated, trade secret, IP, legal, financial, human/non-human-readable).
2. Apply **data classifications** (public, private, sensitive, confidential, restricted, critical).
3. Explain the three **data states** (at rest, in transit, in use) and **sovereignty/geolocation**.
4. Compare **methods to secure data**: encryption, hashing, masking, tokenization, obfuscation, segmentation, permission restrictions, geographic restrictions.

---

## 4. Concept Overview

Data is the asset security exists to protect. This objective is about **classifying** data (so you know how to handle it), understanding its **states** and **location** constraints (sovereignty), and applying the right **protection method** to each. CompTIA tests it as comparison: given data of a certain type/state, which protection method or classification applies?

Four clusters:

1. **Data types** — what *kind* of data it is (drives obligations).
2. **Classifications** — labels by sensitivity (drive handling rules).
3. **States & location** — at rest/in transit/in use; sovereignty/geolocation.
4. **Methods** — encryption, hashing, masking, tokenization, obfuscation, segmentation, permissions, geographic restrictions.

## 6. Deep Technical Explanation

### Data types

| Type | Meaning |
|------|---------|
| **Regulated** | Governed by law (PII, PHI/HIPAA, PCI cardholder data) — strict handling |
| **Trade secret** | Confidential competitive info (formulas, processes) |
| **Intellectual property** | Patents, copyrights, source code, designs |
| **Legal information** | Contracts, litigation, legal records |
| **Financial information** | Accounts, statements, transactions |
| **Human-readable** | Directly readable by people (plaintext) |
| **Non-human-readable** | Needs decoding/processing (binary, encrypted, encoded) |

### Data classifications (by sensitivity)

| Classification | Meaning |
|----------------|---------|
| **Public** | No harm if disclosed (marketing, public docs) |
| **Private** | Personal data; disclosure harms individuals |
| **Sensitive** | Requires protection; disclosure causes harm |
| **Confidential** | Restricted to authorized parties; significant harm if leaked |
| **Restricted** | Highly controlled; severe harm if disclosed |
| **Critical** | Essential data whose loss/disclosure is catastrophic |

> Classification drives handling: **public** can be shared freely; **restricted/critical/confidential** need strong controls (encryption, tight permissions). The exact tiers vary by org, but CompTIA expects you to rank by sensitivity and pick controls accordingly.

### Data states (the three states — each needs different protection)

| State | What it is | How to protect |
|-------|-----------|----------------|
| **At rest** | Stored (disk, DB, backup) | **Encryption** (full-disk, DB), access control |
| **In transit** | Moving over a network | **TLS/IPSec** (encrypt the channel) |
| **In use** | Being processed in memory | Hardest — access control, secure enclaves, memory protection |

> **All three states need protection.** Encrypting at rest *and* in transit is standard; **in use** is the hardest (data must be decrypted to process). "Data being sent over the internet" = in transit → TLS; "data on a hard drive" = at rest → disk encryption.

### Data sovereignty & geolocation

- **Data sovereignty** — data is subject to the **laws of the country where it's physically stored** (e.g., EU data under GDPR). Critical for cloud, where data may live anywhere.
- **Geolocation** — knowing/controlling *where* data and users are; drives **geographic restrictions** (geofencing) to keep data in compliant regions.

> This ties to compliance (5.4): storing EU residents' data outside approved regions can violate sovereignty laws.

### Methods to secure data

| Method | What it does | Reversible? | Typical use |
|--------|--------------|-------------|-------------|
| **Encryption** | Encodes data with a key | Yes (with key) | Protect at rest/in transit |
| **Hashing** | One-way digest | **No** | Integrity, password storage |
| **Masking** | Hides part of the data | Partial/no | Display (show last 4 of a card) |
| **Tokenization** | Swaps data for a token (vault maps back) | Yes (via vault) | PCI/payment data |
| **Obfuscation** | Makes data hard to interpret | Varies | Hide logic/data |
| **Segmentation** | Isolates data into zones | n/a | Limit access/blast radius |
| **Permission restrictions** | Controls who can access | n/a | Access control / least privilege |
| **Geographic restrictions** | Limits where data lives/is accessed | n/a | Sovereignty/compliance |

> **Key distinctions:**
> - **Encryption vs. hashing:** encryption is **reversible** (with a key); hashing is **one-way** (irreversible) — used for integrity/passwords.
> - **Masking vs. tokenization:** masking **hides** part of the value (often irreversibly, for display — `****1234`); tokenization **replaces** the value with a token that can be mapped back via a secure vault (common for PCI). Tokenization removes sensitive data from the environment entirely; masking just obscures it on screen.

## 7. Why It Exists

You can't protect data uniformly — a public brochure and a customer's SSN need very different handling. This objective exists so you **classify** data, understand **where and in what state** it lives, and apply the **right protection** (and the right *amount*). It connects directly to compliance (regulated data, sovereignty) and to the CIA triad (confidentiality especially).

---

## 8. How It Works (enterprise)

Organizations **classify** data (labels/tags), then enforce handling rules per class: encryption at rest and in transit, tokenization/masking for payment/PII, permission restrictions (least privilege), and segmentation. **DLP** enforces classification (blocks sensitive data from leaving). **Data sovereignty/geolocation** controls (region pinning, geofencing) keep regulated data in compliant locations. Discovery/inventory (5.4) finds the data to protect.

---

## 9. Advantages

- **Right-sized protection** — controls match data sensitivity.
- **Compliance** — regulated data and sovereignty handled correctly.
- **Reduced breach impact** — tokenized/encrypted data is useless if stolen.
- **Clear handling rules** — classification removes guesswork.

---

## 10. Disadvantages / Challenges

- **Classification effort** — labeling all data is hard and ongoing.
- **In-use protection is difficult** (data must be decrypted to process).
- **Key/vault management** — encryption and tokenization depend on it.
- **Sovereignty complexity** — multi-region cloud makes location hard to control.
- **Over/under-classification** — wrong labels = wasted effort or exposure.

---

## 11. Real-World Examples

- **Tokenization:** a retailer replaces card numbers with tokens so its systems never store real PANs (PCI).
- **Masking:** a support screen shows only the last 4 digits of an SSN.
- **Encryption at rest/in transit:** full-disk encryption on laptops + TLS for web traffic.
- **Data sovereignty:** an EU company pins customer data to EU regions (GDPR).
- **Hashing:** passwords stored as salted hashes (never recoverable).
- **Classification:** a "Confidential" tag triggers DLP to block external email.

---

## 12. Enterprise Usage (SOC relevance)

- **DLP alerts** (sensitive data leaving) are core SOC events tied to classification.
- **Data state** tells you where to look: in-transit anomalies (network), at-rest exposure (storage), in-use (endpoint/memory).
- **Sovereignty/geolocation** violations may surface as alerts (data in the wrong region).
- Knowing **tokenization/masking** helps you assess breach impact (was real data exposed or just tokens?).

---

## 13. Security Implications

- **Unclassified data** gets handled inconsistently → leaks.
- **Unencrypted at rest/in transit** = readable if stolen/intercepted.
- **In-use data** is the hardest to protect (a real exposure window).
- **Sovereignty violations** carry legal/compliance penalties.
- **Reversible vs. irreversible** matters: hashed/tokenized data limits breach impact.

---

## 14. Attacker / Adversary Perspective

- Attackers target **unencrypted at-rest data** (databases, backups) and **in-transit** traffic (no TLS).
- They prize **regulated/financial/IP** data (highest resale/extortion value).
- They look for **data in use** (memory scraping — e.g., POS malware grabbing card data before tokenization).
- **Mis-located data** (sovereignty gaps) and **over-permissioned** stores are soft targets.

---

## 15. Defender Perspective

- **Classify data** and enforce handling per class.
- **Encrypt at rest and in transit;** protect in-use where possible (enclaves).
- **Tokenize/mask** sensitive fields (PII/PCI).
- **Restrict permissions** (least privilege) and **segment** sensitive data.
- **Enforce geographic restrictions** for sovereignty.
- **Use DLP** to enforce classification.

---

## 16. Detection Techniques

- **DLP** for sensitive-data movement (enforces classification).
- **Data discovery/classification tools** to find and label data.
- **Encryption/config scanning** (unencrypted stores).
- **Cloud region/geolocation monitoring** for sovereignty.
- **Access monitoring** on sensitive data stores.

---

## 17. Mitigation Techniques

- **Encryption** (at rest + in transit), **tokenization/masking** (PII/PCI).
- **Hashing** for passwords/integrity.
- **Permission restrictions** (least privilege) + **segmentation**.
- **Geographic restrictions / region pinning** (sovereignty).
- **DLP** + classification enforcement.

---

## 18. Best Practices

1. **Classify data** and apply handling rules by class.
2. **Encrypt at rest and in transit;** protect in-use where feasible.
3. **Tokenize or mask** sensitive fields (PII, payment data).
4. **Use hashing** (with salt) for passwords/integrity — never reversible storage.
5. **Apply least-privilege permissions** and **segment** sensitive data.
6. **Respect data sovereignty** with geographic restrictions.
7. **Deploy DLP** to enforce classification automatically.

## 20. Diagrams

**Three data states:**

```
   DATA AT REST  ─► stored on disk/DB     → encrypt (disk/DB)
   DATA IN TRANSIT ─► moving on network   → encrypt channel (TLS/IPSec)
   DATA IN USE   ─► processing in memory  → access control / enclaves (hardest)
```

**Masking vs. tokenization:**

```
   MASKING:       4111-1111-1111-1234  →  ****-****-****-1234   (hidden for display)
   TOKENIZATION:  4111-1111-1111-1234  →  TKN-9F2A-...          (replaced; vault maps back)
```

**Encryption vs. hashing:**

```
   ENCRYPTION:  plaintext ⇄ ciphertext   (REVERSIBLE with key)
   HASHING:     input → digest           (ONE-WAY, irreversible)
```

## 21. Exam Tips

- **Three data states: at rest (stored), in transit (moving), in use (processing).** Each needs protection; in-use is hardest.
- **Encryption = reversible (with key); hashing = one-way (irreversible).**
- **Masking = hide part for display (`****1234`); tokenization = replace with a token (vault maps back, removes real data).**
- **Data sovereignty = data follows the laws of the country it's stored in.**
- **Geographic restrictions / geofencing** enforce where data lives (compliance).
- **Regulated data** = PII/PHI/cardholder (legal obligations).
- **Classify from public → private → sensitive → confidential → restricted → critical.**
- **In transit → TLS/IPSec; at rest → disk/DB encryption.**
- **Permission restrictions = least-privilege access; segmentation = isolate data.**

1. *"Data moving across the internet should be protected using…"* → Encryption in transit (TLS/IPSec).
2. *"Which method replaces a credit card number with a surrogate value mapped in a secure vault?"* → Tokenization.
3. *"A screen shows only the last four digits of an account number."* → Masking.
4. *"Which is irreversible and used to store passwords?"* → Hashing.
5. *"Data stored in a country is subject to that country's laws. This is…"* → Data sovereignty.
6. *"PII and PHI governed by law are an example of which data type?"* → Regulated data.
7. *"Which data state is the most difficult to protect?"* → In use.
8. *"Restricting which regions data can be stored in addresses…"* → Geographic restrictions / sovereignty.

**Why distractors fail:** they swap encryption/hashing (reversible vs. not), masking/tokenization (hide vs. replace), or the data states. Anchor on reversibility, replace-vs-hide, and stored/moving/processing.

## 23. Interview Notes

- *"Encryption vs. hashing?"* — Encryption is reversible with a key (confidentiality); hashing is one-way (integrity, passwords).
- *"Masking vs. tokenization?"* — Masking hides part of the value for display; tokenization replaces it with a token, removing the real data from systems (vault maps back).
- *"What are the three data states?"* — At rest, in transit, in use; in use is hardest to protect.
- *"What is data sovereignty?"* — Data is subject to the laws of the country where it's physically stored.
- *"Why classify data?"* — To apply the right level of protection and meet compliance.

---

## 24. Common Beginner Mistakes

1. **Confusing encryption (reversible) and hashing (one-way).**
2. **Mixing masking (hide) and tokenization (replace).**
3. **Forgetting "in use" as a data state** (and that it's hardest).
4. **Ignoring data sovereignty** in cloud/multi-region.
5. **Thinking only at-rest needs encryption** (in transit does too).
6. **Treating all data the same** (no classification).
7. **Confusing geographic restrictions with segmentation.**

## 29. Practice Questions

**Q7.** Which data state is generally the MOST difficult to protect?
A. At rest  B. In transit  C. In use  D. Archived

**Q8.** Which is the KEY difference between encryption and hashing?
A. Encryption is one-way; hashing is reversible
B. Encryption is reversible with a key; hashing is one-way and irreversible
C. They are identical
D. Hashing requires a public key

**Q9.** Restricting the regions where data may be stored to comply with local laws is an example of:
A. Geographic restrictions  B. Tokenization  C. Hashing  D. Load balancing

**Q10.** Which classification would apply to data freely shareable with no harm if disclosed?
A. Restricted  B. Confidential  C. Public  D. Critical

### Answer Key & Explanations

- **Q1 — C (Tokenization).** Replaces data with a token; vault maps back.
- **Q2 — B.** In transit → encrypt the channel (TLS/IPSec).
- **Q3 — C (Hashing).** One-way; used for passwords.
- **Q4 — B (Masking).** Hides part of the value for display.
- **Q5 — B (Data sovereignty).** Data follows the host country's laws.
- **Q6 — C (Regulated data).** Governed by law.
- **Q7 — C (In use).** Data must be decrypted to process.
- **Q8 — B.** Encryption is reversible (key); hashing is one-way.
- **Q9 — A (Geographic restrictions).** Limits where data resides.
- **Q10 — C (Public).** No harm if disclosed.
