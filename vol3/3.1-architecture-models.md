## 3. Learning Outcomes

After this objective you will be able to:

1. Compare major architecture models: **cloud, on-premises, hybrid, centralized vs. decentralized**.
2. Explain the **cloud responsibility matrix** and third-party/hybrid considerations.
3. Describe **IaC, serverless, microservices, containerization, virtualization, SDN**, and their security implications.
4. Explain specialized systems: **IoT, ICS/SCADA, RTOS, embedded** — and why they're hard to secure.
5. Apply the **considerations** (availability, scalability, patch availability, inability to patch, cost, etc.) when comparing models.

---

## 4. Concept Overview

**Security architecture** is about *how* you build and arrange systems — and every architecture choice has security trade-offs. This objective is a broad survey: cloud vs. on-prem, modern app patterns (serverless, microservices, containers), networking (SDN, segmentation, air-gap), and specialized/hard-to-patch systems (IoT, ICS/SCADA, embedded, RTOS).

CompTIA tests this as **comparison**: given a model, what are its security implications? The recurring theme is the **shared responsibility** of cloud, the **patchability** problem of specialized systems, and the trade-offs captured in the official **considerations** list (availability, scalability, cost, patch availability, inability to patch, etc.).

## 6. Deep Technical Explanation

### Cloud, on-premises, hybrid

- **Cloud** — provider-hosted, elastic, pay-as-you-go. Security is **shared** (see matrix). Pros: scalability, cost flexibility, **risk transference**, ease of deployment. Cons: less direct control, internet exposure, dependency on the provider.
- **On-premises** — you own and control everything (and all the responsibility). Pros: full control, data locality. Cons: high cost, you handle all patching/scaling/recovery.
- **Hybrid** — mixes both; flexible but adds **integration complexity** and a larger, more varied attack surface.

**Cloud responsibility matrix (shared responsibility):**

| Model | Provider secures | Customer secures |
|-------|------------------|------------------|
| **IaaS** | Physical, network, hypervisor | OS, apps, data, IAM |
| **PaaS** | + OS, runtime | Apps, data, IAM |
| **SaaS** | + apps | Data, access/config |

> **Key cloud rule:** *the customer is ALWAYS responsible for their data and access management.* The provider never "owns" your data security. The lower-level the service (IaaS), the more *you* secure; the higher (SaaS), the less. **Third-party vendors** (5.3) and **hybrid considerations** add shared-responsibility complexity.

### Modern application architectures

- **Infrastructure as code (IaC)** — define infrastructure in code (Terraform, CloudFormation). Pros: consistent, repeatable, version-controlled, fewer manual misconfigs. Risk: a flaw in the code template propagates everywhere; secrets in code.
- **Serverless** — run functions without managing servers. Pros: no server patching (provider handles it), auto-scaling. Risks: less visibility/control, function-level misconfig, vendor lock-in.
- **Microservices** — many small independent services vs. one monolith. Pros: scalable, resilient (one service failing ≠ whole app down), independently deployable. Risks: larger attack surface (many APIs), complex inter-service security.
- **Containerization** — lightweight, share the host OS kernel (Docker). Pros: portable, efficient, fast. Risks: weaker isolation than VMs (shared kernel), insecure images, orchestration (Kubernetes) misconfig.
- **Virtualization** — full VMs with a hypervisor; stronger isolation than containers but heavier. Risk: **VM escape**, sprawl, resource reuse (2.3).

> **Containers vs. VMs:** containers share the host **kernel** (lightweight, less isolation); VMs each have a full OS (heavier, stronger isolation).

### Network infrastructure

- **Physical isolation / air-gapped** — no network connection at all; the strongest isolation (used for highly sensitive/critical systems). Downside: hard to update; not immune (USB/insider).
- **Logical segmentation** — VLANs/subnets divide a network in software (cheaper, flexible, but a misconfig can break isolation).
- **Software-defined networking (SDN)** — separates the **control plane** (decisions) from the **data plane** (forwarding), managed centrally by software. Pros: agile, programmable, centralized policy. Risk: the central controller is a high-value target.

### Centralized vs. decentralized

- **Centralized** — one control point: easier management/visibility, but a **single point of failure** and a concentrated target.
- **Decentralized** — distributed: more resilient (no single point), but harder to manage/secure consistently.

### Specialized systems (the "hard to patch" group)

| System | What it is | Security challenge |
|--------|-----------|--------------------|
| **IoT** | Networked everyday/embedded devices | Weak default security, rarely patched, huge numbers |
| **ICS/SCADA** | Industrial/utility control systems | Legacy, can't take downtime, often unpatchable, high impact |
| **RTOS** | Real-time OS (timing-critical) | Limited security features; can't interrupt for patching |
| **Embedded** | Special-purpose computer in a device | Fixed function, hard/no patching, long lifecycle |

> **The common thread: limited or no patchability.** IoT/ICS/SCADA/RTOS/embedded often **can't be patched** (no downtime allowed, no vendor updates, fixed firmware) — so they're defended by **isolation/segmentation/air-gap** and monitoring rather than patching. "Inability to patch" is an official consideration precisely because of these.

### High availability

Designing for minimal downtime (redundancy, load balancing, clustering, failover). Detailed in **3.4** — here it's one architecture consideration (availability + resilience).

### The "Considerations" (how to compare any model)

The official decision factors when choosing an architecture:

| Consideration | Question it answers |
|---------------|---------------------|
| **Availability** | How much uptime? |
| **Resilience** | How well does it survive failures? |
| **Cost** | How expensive? |
| **Responsiveness** | How fast does it react/perform? |
| **Scalability** | Can it grow on demand? |
| **Ease of deployment** | How quickly can you stand it up? |
| **Risk transference** | Can risk be shifted (e.g., to a cloud provider/insurer)? |
| **Ease of recovery** | How fast to restore after failure? |
| **Patch availability** | Are patches available/easy to apply? |
| **Inability to patch** | What if it *can't* be patched (IoT/ICS)? |
| **Power** | Power needs/constraints? |
| **Compute** | Processing capacity available? |

## 7. Why It Exists

Architecture decisions are made early and are expensive to change — and they bake in security strengths and weaknesses. This objective exists so you can **reason about the trade-offs**: cloud buys scalability and risk transference but adds shared responsibility; air-gap buys isolation but loses convenience; IoT/ICS deliver function but resist patching. Security architecture is choosing the right model for the requirement and compensating for its weaknesses.

---

## 8. How It Works (enterprise)

Architects pick models against the **considerations** and the **shared-responsibility matrix**. Cloud workloads use IaC for consistent, auditable deployments; microservices/containers are orchestrated (Kubernetes) with image scanning and network policy; SDN centralizes network policy; sensitive/critical systems (ICS/SCADA, classified) are air-gapped or tightly segmented and monitored because they can't be patched normally. Each choice maps to controls (CSPM for cloud, segmentation for OT, hardening for embedded).

---

## 9. Advantages (model-dependent)

- **Cloud:** scalability, cost flexibility, risk transference, fast deployment.
- **IaC:** consistency, version control, fewer manual misconfigs.
- **Microservices/containers:** scalability, resilience, portability.
- **Air-gap:** strongest isolation.
- **SDN:** centralized, programmable network policy.

---

## 10. Disadvantages (model-dependent)

- **Cloud:** shared responsibility confusion, less control, internet exposure.
- **Containers:** weaker isolation (shared kernel), image risks.
- **IoT/ICS/RTOS/embedded:** poor/no patchability, weak built-in security, high impact.
- **Centralized:** single point of failure/target.
- **SDN:** controller is a high-value single target.

---

## 11. Real-World Examples

- **SaaS responsibility gap:** a company assumes the SaaS provider secures their data config — it doesn't; a public sharing setting leaks data.
- **IaC misconfig:** a Terraform template opens a security group to 0.0.0.0/0 across all environments.
- **Container escape:** a misconfigured container reaches the host.
- **ICS/SCADA:** a utility's unpatchable legacy controllers, protected only by air-gap/segmentation.
- **IoT botnet:** thousands of default-credential cameras conscripted into a DDoS botnet.

---

## 12. Enterprise Usage (SOC relevance)

- **Cloud monitoring (CSPM/CloudTrail):** misconfig and access alerts dominate cloud SOC work.
- **Container/orchestration alerts:** image vulnerabilities, anomalous container behavior.
- **OT/ICS monitoring:** specialized; you watch for any unexpected change because these can't be patched.
- Understanding the **responsibility matrix** tells you *what you're even responsible for monitoring* in cloud.
- **IoT** shows up as unmanaged, noisy assets on the network.

---

## 13. Security Implications

- **Misunderstanding shared responsibility** is a top cloud breach cause (unsecured data/buckets).
- **Specialized systems** (IoT/ICS) can't be patched → must be isolated/segmented and monitored.
- **Containers** need image scanning and good isolation config.
- **Centralized controllers (SDN)** are single high-value targets.
- **IaC** flaws scale instantly — secure the templates and secrets.

---

## 14. Attacker / Adversary Perspective

- Attackers hunt **cloud misconfigurations** (open storage, over-permissive IAM) — fast, common wins.
- They target **IoT** for botnets (default creds, no patches).
- **ICS/SCADA** are prized by nation-states (critical-infrastructure impact, hard to defend).
- They abuse **insecure container images** and exposed orchestration APIs.
- **IaC/secrets in code** leak credentials at scale.

---

## 15. Defender Perspective

- **Know your responsibility** (matrix) and secure *your* part (data, IAM, config) in cloud.
- **Isolate/segment** unpatchable systems (IoT/ICS/embedded); monitor them closely.
- **Scan IaC and container images;** manage secrets properly.
- **Harden orchestration and SDN controllers.**
- **Use the considerations** to pick models whose weaknesses you can compensate for.

---

## 16. Detection Techniques

- **CSPM / cloud config scanning** for misconfigurations.
- **Container/image scanning** and runtime monitoring.
- **OT/ICS network monitoring** (passive, change-detection).
- **IoT discovery + network behavior analytics.**
- **IaC scanning** (Checkov/tfsec) for template flaws.

---

## 17. Mitigation Techniques

- **Cloud:** least-privilege IAM, secure config baselines, encryption, CSPM.
- **Specialized systems:** segmentation/air-gap, compensating controls (since patching is limited).
- **Containers:** minimal/scanned images, namespaces, network policies.
- **SDN/centralized:** harden and protect the controller; redundancy.
- **IaC:** code review, secret management, policy-as-code.

---

## 18. Best Practices

1. **Apply the shared-responsibility matrix** — secure your data/IAM in cloud.
2. **Segment/air-gap unpatchable systems** (IoT/ICS/embedded/RTOS).
3. **Scan IaC and container images;** never hardcode secrets.
4. **Use IaC** for consistent, auditable, repeatable deployments.
5. **Harden centralized controllers** (SDN) and add redundancy.
6. **Choose models via the considerations** and compensate for weaknesses.
7. **Monitor everything**, especially systems you can't patch.

## 22. Common CompTIA Question Styles

1. *"In a SaaS model, who is responsible for securing the customer's data and access?"* → The customer.
2. *"Which provides the strongest isolation for a highly sensitive system?"* → Air-gapped (physical isolation).
3. *"What separates the network control plane from the data/forwarding plane?"* → SDN.
4. *"Why are ICS/SCADA systems difficult to secure?"* → They're legacy, can't take downtime, and often can't be patched.
5. *"Which is lighter but shares the host OS kernel, offering weaker isolation?"* → Containers.
6. *"Managing infrastructure through version-controlled definition files is…"* → Infrastructure as code (IaC).
7. *"A single SDN controller failing takes down network control. This reflects which risk?"* → Single point of failure (centralized).
8. *"Which consideration applies when a device simply cannot be updated?"* → Inability to patch.

**Why distractors fail:** they blur the responsibility matrix, swap containers/VMs, or confuse air-gap with logical segmentation. Anchor on *who secures what*, *kernel sharing*, and *patchability*.

## 23. Interview Notes

- *"Explain the cloud shared responsibility model."* — Provider secures lower layers (more in SaaS, less in IaaS); customer always secures their data, access, and configuration.
- *"Containers vs. VMs?"* — Containers share the host kernel (lightweight, less isolation); VMs run full OSes (stronger isolation, heavier).
- *"Why are ICS/SCADA hard to secure?"* — Legacy, uptime-critical, often unpatchable, high impact — defended by segmentation/air-gap and monitoring.
- *"What is IaC and its security benefit/risk?"* — Infrastructure defined as code; consistent and auditable, but template flaws and embedded secrets scale.
- *"What is SDN?"* — Software-controlled networking separating the control plane from the data plane.

## 24. Common Beginner Mistakes

1. **Thinking the cloud provider secures your data** (it never does fully — data/IAM are yours).
2. **Confusing containers and VMs** (kernel sharing vs. full OS).
3. **Mixing air-gap (physical) with logical segmentation (software).**
4. **Assuming IoT/ICS can be patched like normal IT** (often they can't).
5. **Forgetting SDN's control/data plane split.**
6. **Overlooking centralized = single point of failure.**
7. **Ignoring "inability to patch" as a real architecture consideration.**

## 29. Practice Questions

**Q1.** In a SaaS deployment, which security responsibility ALWAYS remains with the customer?
A. Patching the operating system  B. Securing the data and managing access  C. Maintaining the hypervisor  D. Physical datacenter security

**Q2.** Which architecture provides the STRONGEST isolation for a highly sensitive system?
A. Logical segmentation  B. Containerization  C. Air-gapped (physical isolation)  D. Microservices

**Q3.** Containers differ from virtual machines primarily because containers:
A. Each run a full guest operating system
B. Share the host operating system kernel, making them lighter but less isolated
C. Provide stronger isolation than VMs
D. Cannot be deployed in the cloud

**Q4.** Why are ICS/SCADA systems particularly difficult to secure?
A. They are brand new and untested
B. They are often legacy, uptime-critical, and cannot be easily patched
C. They have no network connectivity ever
D. They run only modern operating systems

**Q5.** Managing infrastructure through version-controlled, machine-readable definition files is called:
A. Serverless  B. Infrastructure as code (IaC)  C. Microservices  D. SDN

**Q6.** Software-defined networking (SDN) is characterized by:
A. Combining the control and data planes into hardware
B. Separating the control plane (decisions) from the data plane (forwarding)
C. Air-gapping all switches
D. Eliminating the need for routers

**Q7.** Which is a key DISADVANTAGE of a centralized architecture?
A. It is too resilient  B. It creates a single point of failure and a concentrated target  C. It is impossible to manage  D. It cannot scale at all

**Q8.** A device's firmware cannot be updated by the vendor or the customer. Which architecture consideration does this represent?
A. Scalability  B. Risk transference  C. Inability to patch  D. Responsiveness

**Q9.** Which model lets developers run code without managing the underlying servers, with the provider handling infrastructure patching?
A. On-premises  B. Serverless  C. Containerization  D. Air-gapped

**Q10.** Moving certain risks to a cloud provider or insurer is which official consideration?
A. Risk transference  B. Ease of recovery  C. Compute  D. Resilience

### Answer Key & Explanations

- **Q1 — B.** Data and access are always the customer's responsibility in cloud.
- **Q2 — C (Air-gapped).** Physical isolation is strongest.
- **Q3 — B.** Containers share the host kernel (lighter, weaker isolation).
- **Q4 — B.** Legacy, uptime-critical, often unpatchable.
- **Q5 — B (IaC).** Infrastructure defined as code.
- **Q6 — B.** SDN separates control from data plane.
- **Q7 — B.** Centralized = single point of failure/target.
- **Q8 — C (Inability to patch).** The defining consideration for such devices.
- **Q9 — B (Serverless).** No server management; provider patches infra.
- **Q10 — A (Risk transference).** Shifting risk to a third party.
