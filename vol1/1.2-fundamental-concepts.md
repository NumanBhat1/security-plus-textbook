## 4. Concept Overview

Objective 1.2 is the **conceptual bedrock** of the entire exam. Almost every later objective is an application of these ideas. The objective groups several "first principles":

- **CIA** — the three goals security exists to protect.
- **Non-repudiation** — proof that an action can't be denied.
- **AAA** — the lifecycle of identity: prove who you are, get what you're allowed, and be held accountable.
- **Gap analysis** — the method for finding what's missing (ties directly to 1.1's controls).
- **Zero Trust** — the modern access architecture: "never trust, always verify," split into a decision plane and an enforcement plane.
- **Physical security** — protecting the tangible world (an extension of 1.1's *Physical* category).
- **Deception technology** — traps that detect and study attackers.

Think of this objective as the vocabulary and worldview a security professional reasons with daily.

## 5. Definitions

| Term | Definition |
|------|------------|
| **Confidentiality** | Ensuring information is accessible only to authorized parties (achieved via encryption, access control). |
| **Integrity** | Ensuring data is accurate and unaltered by unauthorized parties (achieved via hashing, digital signatures). |
| **Availability** | Ensuring systems/data are accessible when needed (achieved via redundancy, backups, DDoS protection). |
| **Non-repudiation** | Assurance that a party cannot deny the authenticity/origin of their action (provided by digital signatures, logging). |
| **Authentication** | Proving an identity is who/what it claims to be. |
| **Authorization** | Determining what an authenticated identity is permitted to do. |
| **Accounting** | Recording and tracking what an identity did (logs, audit trails). |
| **Gap analysis** | Comparing the current security state against a desired/required state to identify deficiencies. |
| **Zero Trust** | A security model that assumes no implicit trust; every request is continuously verified regardless of network location. |
| **Control Plane (ZT)** | The "decision-making" layer of Zero Trust — defines and decides policy. |
| **Data Plane (ZT)** | The "enforcement" layer — where access decisions are applied to actual traffic. |
| **Policy Engine (PE)** | The Zero Trust component that *decides* to grant/deny access based on policy. |
| **Policy Administrator (PA)** | The component that *executes* the PE's decision (establishes/tears down the connection). |
| **Policy Enforcement Point (PEP)** | The gateway in the data plane that *enforces* the decision on the actual session. |
| **Honeypot** | A decoy *system* designed to lure and observe attackers. |
| **Honeynet** | A *network* of honeypots simulating a realistic environment. |
| **Honeyfile** | A decoy *file* that triggers an alert when accessed. |
| **Honeytoken** | A decoy *piece of data* (fake credential, API key, record) that signals compromise when used. |

## 6. Deep Technical Explanation

### The CIA Triad — the "why" of all security

Every security control ultimately serves one or more of these three goals:

```
                    ┌─────────────────┐
                    │   THE CIA TRIAD  │
                    └─────────────────┘
            Confidentiality   Integrity   Availability
            (keep secret)     (keep true) (keep usable)
                   │              │             │
       Encryption, ACLs    Hashing, digital   Redundancy,
       MFA, masking        signatures, FIM    backups, DDoS
                                              protection, HA
```

- **Confidentiality** is broken by *disclosure* (data leaks, shoulder surfing, sniffing).
- **Integrity** is broken by *alteration* (tampering, MITM modification, corruption).
- **Availability** is broken by *destruction/denial* (DDoS, ransomware, hardware failure).

Sometimes you'll see **non-repudiation** added as a fourth goal, and the **DAD triad** (Disclosure, Alteration, Destruction/Denial) as CIA's inverse — what attackers *do* to each pillar.

> **Trade-offs:** the pillars can conflict. Heavy encryption (confidentiality) can slow systems (availability). Strict integrity checks add overhead. Security design is constant balancing among the three based on what the asset needs most.

### Non-repudiation

Non-repudiation means a party **cannot credibly deny** having performed an action. The classic mechanism is the **digital signature**: the sender signs data with their *private key*; anyone can verify with the *public key*. Because only the sender holds the private key, they cannot deny signing. Logging/audit trails (accounting) also support non-repudiation. (Cryptographic detail is in Objective 1.4.)

> **Key distinction:** integrity says "the data wasn't changed"; non-repudiation says "*you* are the one who sent/signed it, and you can't deny it." A digital signature provides *both* integrity and non-repudiation.

### AAA — the identity lifecycle

```
   AUTHENTICATION → AUTHORIZATION → ACCOUNTING
   "Who are you?"   "What can you   "What did you
                     do?"            do?"
   (verify identity) (grant rights)  (log actions)
```

- **Authenticating people:** something you *know* (password), *have* (token/phone), *are* (biometric), plus location/behavior. Multifactor = two or more *different* factors. (Detailed in 4.6.)
- **Authenticating systems:** machines/services prove identity too — via **certificates** (mutual TLS), **device certificates**, **802.1X** port authentication, API keys, and shared secrets. A server proving its identity with a TLS certificate is system authentication.
- **Authorization models** (the rules for *what you can do*) — detailed table below; full treatment in 4.6:
  - **DAC** (Discretionary): owner sets permissions.
  - **MAC** (Mandatory): system enforces labels/clearances (military-style).
  - **RBAC** (Role-Based): permissions by job role.
  - **Rule-based**: conditions/rules (e.g., time-of-day) decide access.
  - **ABAC** (Attribute-Based): policies evaluate attributes (user, resource, environment).

### Gap analysis

A gap analysis answers *"Where are we now vs. where we need to be?"*

```
   CURRENT STATE ──────► [ GAP ] ──────► DESIRED STATE
   (existing controls)              (required by framework/policy/risk)
                  └── the gap = what must be added/fixed ──┘
```

It's the bridge between Objective 1.1 (controls) and the rest of the program: you list required controls (from a framework like NIST/ISO or your risk assessment), compare them to what you actually have, and the *delta* is your remediation roadmap. Gap analysis is common before audits, after acquisitions, and during framework adoption.

### Zero Trust — "never trust, always verify"

Traditional security used a *perimeter* model: trusted "inside," untrusted "outside." Zero Trust **eliminates implicit trust** — being on the internal network grants nothing. Every request is authenticated, authorized, and continuously evaluated.

CompTIA splits Zero Trust into two planes:

```
   ┌──────────────────────── CONTROL PLANE (decides) ───────────────────────┐
   │   Adaptive identity → Threat scope reduction → Policy-driven access     │
   │                                                                         │
   │     ┌──────────────┐         decision         ┌───────────────────┐     │
   │     │ Policy Engine│ ───────────────────────► │ Policy            │     │
   │     │   (decides)  │                          │ Administrator     │     │
   │     └──────────────┘                          │ (executes decision)│    │
   │                                               └─────────┬─────────┘     │
   └─────────────────────────────────────────────────────────│──────────────┘
                                                              │ configures
   ┌──────────────────────── DATA PLANE (enforces) ──────────▼──────────────┐
   │   Subject/System  ──►  Policy Enforcement Point (PEP)  ──►  Resource     │
   │                        (the gateway that allows/blocks the session)      │
   │   Implicit trust zones (segments where, once admitted, you can act)      │
   └─────────────────────────────────────────────────────────────────────────┘
```

**Control Plane components (the decision side):**
- **Policy Engine (PE):** the brain — evaluates policy and *decides* grant/deny.
- **Policy Administrator (PA):** the hands — *executes* the PE's decision by creating/closing the session and issuing credentials/tokens.
- **Adaptive identity:** identity decisions adjust based on context/risk (location, device posture, behavior).
- **Threat scope reduction:** shrink what any identity can reach (least privilege, micro-segmentation) so a compromise has limited blast radius.
- **Policy-driven access control:** access is governed by centrally-defined policy, not network location.

**Data Plane components (the enforcement side):**
- **Subject/System:** the user or machine requesting access.
- **Policy Enforcement Point (PEP):** the gateway that actually *enforces* the decision on the live session (allows/blocks/monitors traffic).
- **Implicit trust zones:** defined segments where, once you're admitted, communication is permitted — kept as small as possible.

> **The single most important ZT exam fact:** **PE decides, PA executes the decision, PEP enforces on the data path.** Control plane = decision; data plane = enforcement.

### Physical security (extends 1.1's Physical category)

| Control | Purpose |
|---------|---------|
| **Bollards** | Short posts that stop vehicles from ramming buildings/entrances. |
| **Access control vestibule** (mantrap) | Two-door airlock; only one door opens at a time → stops tailgating/piggybacking. |
| **Fencing** | Perimeter barrier; deters and delays intruders. |
| **Video surveillance (CCTV)** | Detective control; records and enables review. |
| **Security guard** | Human control; deters, detects, and can respond. |
| **Access badge** | Credential for entry; integrates with logging (detective). |
| **Lighting** | Deters intrusion and improves camera/guard effectiveness. |
| **Sensors** | Detect presence/intrusion (four types below). |

**The four sensor types (high-yield):**

| Sensor | Detects via |
|--------|-------------|
| **Infrared (IR)** | Body **heat** (thermal signatures). |
| **Pressure** | **Weight/force** on a surface (floor mats, etc.). |
| **Microwave** | **Motion** via reflected microwave energy (works through some materials). |
| **Ultrasonic** | **Motion** via reflected sound waves (Doppler shift). |

### Deception and disruption technology

These are *traps* — they have no legitimate purpose, so **any interaction with them is suspicious by definition**, making them excellent low-false-positive detective controls.

| Tech | What it is | Trigger |
|------|-----------|---------|
| **Honeypot** | A single decoy **system** | Attacker probes/attacks it → you observe TTPs. |
| **Honeynet** | A **network** of honeypots | Simulates a realistic environment to study attackers at scale. |
| **Honeyfile** | A decoy **file** (e.g., "passwords.xlsx") | Alert fires when it's opened/copied/exfiltrated. |
| **Honeytoken** | A decoy **data item** (fake API key, credential, DB record) | Use of it anywhere = proof of compromise. |

> **Scale memory hook:** honey**token** (one data item) < honey**file** (one file) < honey**pot** (one system) < honey**net** (many systems).

## 7. Why It Exists

- **CIA** exists because "security" is too vague to act on; the triad gives concrete, measurable goals to design controls around.
- **Non-repudiation** exists because digital actions are easy to deny without proof; commerce, legal, and forensic processes need binding evidence of who did what.
- **AAA** exists to structure identity end-to-end — without it you couldn't grant least privilege or hold anyone accountable.
- **Gap analysis** exists because you can't improve what you haven't measured against a standard.
- **Zero Trust** exists because the perimeter model failed — VPNs, cloud, remote work, and insider threats made "inside = trusted" dangerous. Breaches spread laterally precisely because internal trust was implicit.
- **Physical security** exists because logical controls are meaningless if someone can walk out with the server.
- **Deception tech** exists because detecting a skilled attacker in normal traffic is hard; a trap that *only* an intruder would touch is a clean signal.

---

## 8. How It Works (in an enterprise)

- **CIA** drives control selection: classify each asset by which pillar matters most (a public website prioritizes availability; medical records prioritize confidentiality).
- **AAA** is implemented via identity providers (IdP), directory services (AD/LDAP), RADIUS/TACACS+ for network access, and SSO/federation.
- **Zero Trust** is implemented with an IdP + policy engine (e.g., conditional access), micro-segmentation, device posture checks, and PEPs (proxies/gateways/ZTNA agents) on the data path.
- **Gap analysis** runs periodically and before audits; results feed the risk register (5.2).
- **Physical + deception** controls are deployed in facilities and networks respectively, feeding alerts into the SOC's SIEM.

---

## 9. Advantages

- **CIA:** simple, universal framework; makes security goals concrete and communicable.
- **Non-repudiation:** enables trust in transactions, audit, and legal defensibility.
- **AAA:** enforces least privilege and accountability; foundation of access control.
- **Gap analysis:** turns "are we secure?" into an actionable list.
- **Zero Trust:** dramatically limits lateral movement and blast radius; fits cloud/remote reality.
- **Physical security:** protects against the threats logical controls can't (theft, tampering, destruction).
- **Deception tech:** extremely low false-positive detection; reveals attacker TTPs early.

---

## 10. Disadvantages

- **CIA pillars conflict** — optimizing one can hurt another (security vs. usability/performance).
- **Non-repudiation depends on key security** — if a private key is stolen, non-repudiation breaks.
- **AAA is only as strong as authentication** — weak passwords undermine the whole chain.
- **Gap analysis is a snapshot** — it goes stale; needs repeating.
- **Zero Trust is complex and costly** to implement; can add latency and friction; poorly tuned policies frustrate users.
- **Physical controls cost money and space** and can fail (tailgating defeats badges).
- **Deception tech** must be realistic to fool skilled attackers and carefully isolated so it isn't used as a pivot against you.

---

## 11. Real-World Examples

- **Confidentiality:** TLS on a banking site; full-disk encryption on a laptop.
- **Integrity:** verifying a downloaded ISO's SHA-256 hash (exactly your `hash-checker` tool).
- **Availability:** a load-balanced web farm surviving one server's failure; DDoS scrubbing.
- **Non-repudiation:** a digitally signed email/contract the sender can't later deny.
- **AAA:** logging into a corporate VPN (authenticate) → access only your department's share (authorize) → every action logged (account).
- **Gap analysis:** a company adopting ISO 27001 maps current controls to Annex A and finds it lacks an incident-response policy.
- **Zero Trust:** Microsoft Conditional Access blocking a login from an unmanaged device even with the correct password.
- **Physical:** a data center with bollards, a mantrap vestibule, badge readers, CCTV, and pressure sensors on the raised floor.
- **Deception:** a honeyfile named `Q4_Salaries.xlsx` that pages the SOC the instant it's opened.

---

## 12. Enterprise Usage

A SOC analyst lives inside these concepts:

- You triage alerts by **CIA impact** ("does this threaten confidentiality, integrity, or availability?").
- You use **accounting** (logs) constantly — log analysis *is* the accounting pillar in action.
- You'll see **Zero Trust** signals (impossible travel, device posture failures) as alert sources.
- **Honeytokens/honeyfiles** generate some of the highest-fidelity alerts you'll ever investigate — treat them as near-certain compromise.
- **Gap analysis** output often becomes your detection-engineering backlog ("we have no alerting for X").

---

## 13. Security Implications

- A program that protects confidentiality and integrity but ignores **availability** can be taken down by a simple DoS — all three pillars must be addressed.
- **Implicit trust** (the opposite of Zero Trust) is why breaches spread; flat networks with internal trust let one compromised host reach everything.
- **Physical access usually beats logical controls** — "if I can touch it, I can own it" (boot from USB, steal the disk). Physical security is foundational, not optional.
- **Honeypots must be isolated** — a poorly contained honeypot can become a launch pad into the real network.

---

## 14. Attacker Perspective

- Attackers target whichever **CIA** pillar achieves their goal: steal data (confidentiality), tamper/fraud (integrity), extort via ransomware/DDoS (availability).
- They exploit **implicit trust** to move laterally — Zero Trust is specifically designed to stop them.
- They try to **forge identity** (steal credentials/tokens, bypass MFA) to defeat authentication, and **escalate privileges** to defeat authorization.
- Skilled attackers **avoid known honeypots** and watch for deception, but **honeytokens are hard to avoid** because they look like real loot — using a planted fake credential instantly betrays the intruder.
- Physically, attackers **tailgate** through doors (defeated by vestibules), clone **RFID badges**, and shoulder-surf.

---

## 15. Defender Perspective

- Map every asset and alert to **CIA** to prioritize.
- Build **AAA** with MFA + least privilege + thorough logging.
- Move toward **Zero Trust**: verify explicitly, least-privilege access, assume breach.
- Run regular **gap analyses** to find missing controls (then build them).
- Layer **physical** controls (deter → prevent → detect) around facilities.
- Deploy **deception** widely — honeytokens are cheap, high-signal tripwires. Treat any honeypot/honeytoken hit as a high-severity incident.

---

## 16. Detection Techniques

- **Integrity monitoring:** File Integrity Monitoring (FIM), hash comparison, digital signature verification.
- **Availability monitoring:** uptime/health checks, DDoS detection, capacity alerts.
- **AAA-based detection:** failed logons (Event ID 4625), impossible travel, privilege-change events (4672/4728/4732), anomalous authorization.
- **Zero Trust signals:** device-posture failures, conditional-access denials, anomalous identity behavior.
- **Deception detection:** *any* honeypot/honeynet/honeyfile/honeytoken interaction = alert (lowest false-positive source you have).

---

## 17. Mitigation Techniques

- **Confidentiality:** encryption (at rest/in transit), access control, masking/tokenization, DLP.
- **Integrity:** hashing, digital signatures, FIM, input validation, code signing.
- **Availability:** redundancy, clustering/load balancing, backups, UPS/generators, DDoS protection, capacity planning.
- **Non-repudiation:** digital signatures + protected, time-stamped logs.
- **AAA:** MFA, least privilege, RBAC/ABAC, centralized logging.
- **Zero Trust:** micro-segmentation, continuous verification, least privilege, PEP enforcement.
- **Physical:** vestibules (anti-tailgating), bollards, badges + logging, CCTV, sensors, lighting.

---

## 18. Best Practices

1. **Design to the CIA pillar that matters most** for each asset, but never ignore the other two.
2. **Always use MFA** and least privilege for AAA.
3. **Log everything** — accounting is your forensic lifeline.
4. **Adopt Zero Trust principles** even incrementally: verify explicitly, least privilege, assume breach.
5. **Run gap analyses on a schedule** and after major changes (M&A, cloud migration).
6. **Layer physical controls** and use vestibules where tailgating is a risk.
7. **Deploy honeytokens broadly** — cheap, near-zero false positives.
8. **Isolate honeypots** so they can't be weaponized against you.

## 21. Exam Tips

- **CIA mapping:** "kept secret" = Confidentiality; "unaltered/accurate" = Integrity; "accessible/uptime" = Availability.
- **Hashing → Integrity. Encryption → Confidentiality.** Don't swap them.
- **Digital signature = Integrity + Non-repudiation + Authentication.** If a question wants "prove the sender can't deny it," the answer is non-repudiation (digital signature).
- **Authentication ≠ Authorization.** "Prove identity" vs. "what you're allowed." A classic trap.
- **Zero Trust:** **PE decides, PA executes, PEP enforces.** Control plane = decision; data plane = enforcement.
- **Access control vestibule/mantrap = anti-tailgating.**
- **Sensors:** Infrared=heat, Pressure=weight, Microwave & Ultrasonic=motion (microwave reflects EM, ultrasonic reflects sound).
- **Honeytoken** is the answer when "fake credential/data used to detect misuse." **Honeyfile** for a fake file; **honeypot** for a system; **honeynet** for a network.
- "**Never trust, always verify**" = Zero Trust signature phrase.

## 22. Common CompTIA Question Styles

1. **CIA mapping scenario:** *"An attacker modifies invoice amounts in transit. Which pillar is impacted?"* → Integrity.
2. **Non-repudiation:** *"Which provides proof a user sent a message and prevents denial?"* → Digital signature (non-repudiation).
3. **AAA split:** *"A user logs in successfully but cannot open the finance app. Which AAA component is involved in the denial?"* → Authorization.
4. **Zero Trust components:** *"In a Zero Trust architecture, which component makes the access decision?"* → Policy Engine. *"Which enforces it on the session?"* → PEP.
5. **Physical:** *"Which control best prevents tailgating?"* → Access control vestibule.
6. **Sensors:** *"Which sensor detects body heat?"* → Infrared.
7. **Deception:** *"A fake AWS key is planted in code; its use triggers an alert. What is it?"* → Honeytoken.
8. **Gap analysis:** *"A firm compares its current controls to ISO 27001 requirements to find deficiencies. What is this?"* → Gap analysis.

**Why distractors fail:** they swap confidentiality/integrity, authentication/authorization, or PE/PA/PEP. Anchor on the *defining verb/role* in the scenario.

## 24. Common Beginner Mistakes

1. **Swapping integrity and confidentiality** (hashing vs. encryption).
2. **Confusing authentication with authorization.**
3. Thinking **Zero Trust = a product** — it's an architecture/philosophy, not a single tool.
4. Mixing up **PE / PA / PEP** — remember decide / execute / enforce.
5. Believing being "on the internal network" should grant trust (the *opposite* of Zero Trust).
6. Forgetting **availability** is part of security (it's not just an "ops" concern).
7. Calling every honey-thing a "honeypot" — know token < file < pot < net.
8. Treating non-repudiation as the same as integrity — a signature gives *both*, but they answer different questions.

## 27. Key Facts

- **CIA = Confidentiality, Integrity, Availability** — the three goals of security.
- **Hashing = integrity; Encryption = confidentiality; Redundancy/backups = availability.**
- **Non-repudiation = can't deny it = digital signatures.**
- **AAA = Authentication (who) → Authorization (what) → Accounting (logged).**
- **Authorization models:** DAC, MAC, RBAC, Rule-based, ABAC.
- **Gap analysis = current vs. desired state → find deficiencies.**
- **Zero Trust = never trust, always verify; no implicit trust.** Control plane decides (**PE** decides, **PA** executes); Data plane enforces (**PEP**).
- **Vestibule/mantrap = anti-tailgating; bollards = anti-vehicle.**
- **Sensors:** Infrared=heat, Pressure=weight, Microwave/Ultrasonic=motion.
- **Deception:** honeytoken (data) < honeyfile (file) < honeypot (system) < honeynet (network); any hit = high-fidelity alert.

## 29. Practice Questions

**Q1.** An attacker intercepts and alters the contents of a financial transaction in transit. Which part of the CIA triad is primarily affected?
A. Confidentiality  B. Integrity  C. Availability  D. Non-repudiation

**Q2.** Which mechanism BEST provides non-repudiation?
A. Symmetric encryption  B. Hashing  C. Digital signature  D. Access control list

**Q3.** A user authenticates successfully but is denied access to the payroll system. Which AAA function denied the access?
A. Authentication  B. Authorization  C. Accounting  D. Auditing

**Q4.** In a Zero Trust architecture, which component is responsible for MAKING the access decision?
A. Policy Enforcement Point  B. Policy Administrator  C. Policy Engine  D. Subject

**Q5.** Which Zero Trust component ENFORCES the decision on the live session in the data plane?
A. Policy Engine  B. Policy Administrator  C. Policy Enforcement Point  D. Implicit trust zone

**Q6.** Which physical control is specifically designed to prevent tailgating/piggybacking?
A. Bollards  B. Access control vestibule  C. Fencing  D. Lighting

**Q7.** A motion sensor detects intruders by emitting sound waves and measuring their reflection. Which sensor type is this?
A. Infrared  B. Pressure  C. Microwave  D. Ultrasonic

**Q8.** Security plants a fake set of admin credentials in a config file. When those credentials are used anywhere, an alert fires. What is this called?
A. Honeypot  B. Honeynet  C. Honeyfile  D. Honeytoken

**Q9.** An organization compares its existing controls against the NIST Cybersecurity Framework to identify missing safeguards. This activity is called:
A. Risk assessment  B. Penetration test  C. Gap analysis  D. Attestation

**Q10.** Which is an example of *system* (not people) authentication?
A. Entering a password  B. Scanning a fingerprint  C. A server presenting a TLS certificate  D. Entering an OTP from an app

### Answer Key & Explanations

- **Q1 — B (Integrity).** Altering data in transit attacks integrity. (If they'd *read* it, that'd be confidentiality.)
- **Q2 — C (Digital signature).** Signed with the sender's private key → they can't deny it. Hashing gives integrity only; symmetric encryption doesn't prove a unique signer.
- **Q3 — B (Authorization).** They proved identity (authentication passed) but lacked permission (authorization).
- **Q4 — C (Policy Engine).** The PE decides. (PA executes the decision; PEP enforces.)
- **Q5 — C (Policy Enforcement Point).** The PEP enforces on the live session in the data plane.
- **Q6 — B (Access control vestibule).** A mantrap admits one person at a time, defeating tailgating. Bollards stop vehicles.
- **Q7 — D (Ultrasonic).** Sound-wave reflection = ultrasonic. Microwave uses microwave (EM) reflection; infrared senses heat.
- **Q8 — D (Honeytoken).** A fake data item (credentials) whose use signals compromise. A honeyfile is a fake *file*; a honeypot is a *system*.
- **Q9 — C (Gap analysis).** Current vs. desired/required state to find deficiencies.
- **Q10 — C (TLS certificate).** A machine/service proving identity with a certificate is system authentication; A, B, D authenticate people.

> **Scenario:** You're designing access for a new internal HR application using Zero Trust, and securing the data center that hosts it. Answer the placements.

| Component | Plane | Role (decide/execute/enforce) |
|-----------|-------|-------------------------------|
| Policy Engine | ? | ? |
| Policy Administrator | ? | ? |
| Policy Enforcement Point | ? | ? |

**Part C — Choose the best physical control for each threat:**

| Threat | Best control |
|--------|--------------|
| Vehicle ramming the entrance | ? |

**Answer Key:**
