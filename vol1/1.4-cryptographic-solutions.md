## 4. Concept Overview

Cryptography is the mathematical toolkit that delivers three of security's core goals (from 1.2):

- **Confidentiality** → **encryption** (only authorized parties can read).
- **Integrity** → **hashing** (detect any alteration).
- **Non-repudiation + authentication** → **digital signatures** (prove origin, can't deny).

"**Appropriate** cryptographic solutions" is the key phrase: the exam tests whether you can match the *right* crypto tool to a *requirement*. Encryption ≠ hashing ≠ obfuscation; symmetric ≠ asymmetric; a CRL ≠ OCSP. Choosing wrong (e.g., hashing when you need to recover the data, or a self-signed cert for a public website) is a security failure.

```
   GOAL                CRYPTO TOOL
   Confidentiality  →  Encryption (symmetric/asymmetric)
   Integrity        →  Hashing
   Non-repudiation  →  Digital signature (sign with PRIVATE key)
   Authentication   →  Certificates / asymmetric keys
   Key distribution →  Asymmetric key exchange (e.g., to share a symmetric key)
```

## 5. Definitions

| Term | Definition |
|------|------------|
| **Cryptography** | The practice of securing information using mathematical transformations. |
| **Encryption** | Reversibly transforming plaintext into ciphertext using a key (provides confidentiality). |
| **Symmetric encryption** | One shared secret key encrypts and decrypts (fast). E.g., AES. |
| **Asymmetric encryption** | A key *pair* — public + private; one encrypts, the other decrypts (slower). E.g., RSA, ECC. |
| **Public key** | Shared openly; used to encrypt to you, or to verify your signature. |
| **Private key** | Kept secret; used to decrypt to you, or to create your signature. |
| **PKI (Public Key Infrastructure)** | The framework of CAs, certificates, keys, and policies that manages asymmetric keys and trust. |
| **Key escrow** | A trusted third party stores a copy of keys for recovery (e.g., to decrypt if a key is lost). |
| **Key exchange** | The process of securely agreeing on/sharing a key (e.g., Diffie-Hellman). |
| **Key length** | Number of bits in a key; longer = exponentially harder to brute-force. |
| **Hashing** | A one-way function producing a fixed-length digest; used for integrity (not reversible). |
| **Salt** | Random data added to input before hashing to defeat precomputed (rainbow-table) attacks. |
| **Key stretching** | Deliberately slow hashing (many iterations) to make password cracking expensive (e.g., PBKDF2, bcrypt). |
| **Digital signature** | Data signed with a sender's private key, verifiable with their public key; gives integrity + authentication + non-repudiation. |
| **Obfuscation** | Hiding data's meaning (steganography, tokenization, masking) — *not* the same as encryption. |
| **TPM (Trusted Platform Module)** | A chip on a device's motherboard that stores keys and supports disk encryption/attestation. |
| **HSM (Hardware Security Module)** | A dedicated, tamper-resistant device for generating/storing keys at scale (enterprise). |
| **KMS (Key Management System)** | Software/service to manage the key lifecycle (create, rotate, revoke, store). |
| **Secure enclave** | An isolated, hardware-protected area of a processor for sensitive operations/keys. |
| **Certificate** | A digital document binding a public key to an identity, signed by a CA. |
| **Certificate Authority (CA)** | A trusted entity that issues and signs certificates. |
| **CSR (Certificate Signing Request)** | A request (containing the public key + identity) sent to a CA to obtain a certificate. |
| **CRL (Certificate Revocation List)** | A published list of revoked certificates. |
| **OCSP (Online Certificate Status Protocol)** | A protocol to check a single certificate's status in real time. |
| **Root of trust** | The anchor (root CA) from which all certificate trust derives. |
| **Wildcard certificate** | A cert valid for all subdomains of a domain (`*.example.com`). |
| **Blockchain** | A distributed, append-only ledger of cryptographically linked blocks. |
| **Open public ledger** | A publicly viewable, decentralized record (the transparency property of blockchains). |

## 6. Deep Technical Explanation

### Symmetric vs. Asymmetric (the cornerstone)

```
   SYMMETRIC (one shared key)
   Plaintext --[Key]--> Ciphertext --[same Key]--> Plaintext
   • Fast, efficient for bulk data
   • Problem: how do both sides get the same key securely? (key distribution)
   • Examples: AES (128/192/256), 3DES (legacy), ChaCha20

   ASYMMETRIC (key pair: public + private)
   Encrypt with PUBLIC key  --> Decrypt with PRIVATE key   (confidentiality)
   Sign with PRIVATE key    --> Verify with PUBLIC key      (signatures)
   • Slower; used for small data, key exchange, signatures
   • Solves key distribution (publish the public key freely)
   • Examples: RSA, ECC, Diffie-Hellman (DH)
```

**The hybrid model (how the real world works — e.g., TLS):**

1. Use **asymmetric** crypto to securely **exchange a symmetric session key** (solving key distribution).
2. Use the fast **symmetric** key to encrypt the actual bulk data.

This gives the best of both: asymmetric's secure key distribution + symmetric's speed. Memorize this — it explains TLS, VPNs, and most secure protocols.

**The directional rule (exam-critical):**

| Goal | Use which key |
|------|---------------|
| Encrypt *to* someone (confidentiality) | Their **public** key (only their private key can decrypt) |
| Decrypt what was sent to you | Your **private** key |
| **Sign** (prove it's from you / non-repudiation) | Your **private** key |
| **Verify** a signature | Sender's **public** key |

> Mnemonic: *"Encrypt with the recipient's **public**; sign with your **private**."*

### PKI and key escrow

**PKI** is the whole framework that makes asymmetric crypto usable at scale: certificate authorities, certificates, registration, revocation, and trust hierarchies. **Key escrow** is a deliberate backup of private keys held by a trusted party so encrypted data can be recovered if a key is lost (e.g., an employee leaves) — useful for recovery, but a risk if the escrow is compromised (it's a juicy target).

### Encryption levels

CompTIA wants you to know encryption can be applied at different scopes:

| Level | Scope |
|-------|-------|
| **Full-disk (FDE)** | The entire drive (e.g., BitLocker, LUKS) — protects data at rest if the device is lost/stolen. |
| **Partition** | A specific partition of a disk. |
| **Volume** | A logical volume. |
| **File** | Individual files. |
| **Database** | An entire database (or transparent DB encryption). |
| **Record** | A single record/row/field within a database. |
| **Transport/communication** | Data *in transit* (TLS, IPsec, VPN). |

### Key exchange, algorithms, key length

- **Key exchange:** securely establishing a shared key over an insecure channel. **Diffie-Hellman (DH)** and **ECDH** let two parties derive a shared secret without ever sending it. **Perfect Forward Secrecy (PFS)** uses ephemeral keys so a future key compromise can't decrypt past sessions.
- **Algorithms:** the specific cipher/function. Symmetric: AES, 3DES (legacy). Asymmetric: RSA, ECC, DH. Hashing: SHA-2/SHA-3, MD5 (broken).
- **Key length:** longer keys are exponentially harder to brute-force. AES-256 > AES-128. **ECC achieves equivalent strength with much shorter keys than RSA** (e.g., 256-bit ECC ≈ 3072-bit RSA), making it efficient for mobile/IoT.

### Cryptographic tools

| Tool | What it is | Use |
|------|-----------|-----|
| **TPM** | Chip on a device motherboard | Stores keys, supports full-disk encryption (BitLocker), secure boot, attestation. *Per-device.* |
| **HSM** | Dedicated tamper-resistant appliance | High-volume key generation/storage for enterprises/CAs. *Enterprise-scale.* |
| **KMS** | Key management software/service | Manages key lifecycle: create, store, rotate, revoke (often cloud, e.g., AWS KMS). |
| **Secure enclave** | Isolated hardware region in a CPU | Protects keys/biometrics (e.g., Apple Secure Enclave). |

> **TPM vs. HSM trap:** **TPM = one device** (built into a PC's board); **HSM = enterprise appliance** for many keys at scale. Don't swap them.

### Obfuscation (NOT encryption)

Obfuscation hides meaning but is **not** strong, reversible cryptography:

| Technique | What it does |
|-----------|--------------|
| **Steganography** | Hides data *inside* other data (e.g., a message hidden in an image's pixels). Hides *existence*. |
| **Tokenization** | Replaces sensitive data with a non-sensitive **token**; the real value is stored in a secure vault. Common for card data (PCI). |
| **Data masking** | Shows partial/altered data (e.g., `****-****-****-1234`) so the real value isn't exposed. |

> **Tokenization vs. encryption trap:** tokenization swaps the value for a token with **no mathematical relationship** (lookup in a vault); encryption transforms the value with a key and is reversible *with the key*. Tokenized data can't be reversed without the vault.

### Hashing, salting, key stretching

- **Hashing** is **one-way** (you cannot get the input back) and produces a fixed-length **digest**. Use it for **integrity** (verify a file/download), and for storing passwords (store the hash, not the password). Good: SHA-256/SHA-3. Broken: **MD5, SHA-1** (collision attacks).
- **Salting** adds random data to each input before hashing so identical passwords produce different hashes → defeats **rainbow tables** and makes precomputation useless.
- **Key stretching** runs the hash through many iterations (deliberately slow) so brute-forcing each guess is expensive — **PBKDF2, bcrypt, scrypt, Argon2**. Used for password storage.

> **Hashing properties:** deterministic (same input → same hash), fixed length, one-way, collision-resistant (hard to find two inputs with the same hash).

### Digital signatures (combine hashing + asymmetric)

```
   SIGNING (sender):
     1. Hash the message → digest
     2. Encrypt the digest with sender's PRIVATE key → signature
   VERIFYING (receiver):
     1. Decrypt signature with sender's PUBLIC key → digest A
     2. Hash the received message → digest B
     3. If A == B → integrity intact AND it came from the sender (non-repudiation)
```

A digital signature therefore provides **integrity + authentication + non-repudiation** simultaneously. (It does *not* provide confidentiality — the message itself isn't hidden unless you also encrypt it.)

### Blockchain and open public ledger

- **Blockchain:** a distributed, append-only chain of blocks where each block contains a hash of the previous one. Altering any block changes its hash and breaks the chain → **tamper-evidence/integrity**. Consensus across many nodes means no single party controls it.
- **Open public ledger:** the property that anyone can view the ledger (transparency), as in public cryptocurrencies. Provides decentralization and auditability.

### Certificates (the PKI workhorse — heavily tested)

A **certificate** binds a public key to an identity and is **signed by a CA**, so anyone trusting that CA can trust the binding.

**Lifecycle:**

```
   1. Generate key pair (public + private)
   2. Create a CSR (Certificate Signing Request) — contains public key + identity (NOT the private key)
   3. Submit CSR to a CA
   4. CA validates identity, signs, issues the certificate
   5. Install/use the certificate
   6. If compromised/retired → REVOKE (appears on CRL / checked via OCSP)
```

| Term | Meaning |
|------|---------|
| **CA (Certificate Authority)** | Trusted issuer that signs certs. |
| **CSR** | Request to a CA; carries the **public** key + identity (the private key never leaves you). |
| **CRL** | A *list* of revoked certs the client downloads/checks (can be large, periodic). |
| **OCSP** | A *real-time* query for one cert's status (more efficient than CRL); **OCSP stapling** lets the server present a recent status to avoid client lookups. |
| **Self-signed** | Signed by its own creator, not a trusted CA → browsers warn; fine for internal/testing, **not** for public sites. |
| **Third-party** | Issued by a publicly trusted CA → trusted by browsers/OS automatically. |
| **Root of trust** | The root CA at the top of the chain; trust flows down from it. |
| **Wildcard** | `*.example.com` — covers all first-level subdomains of a domain. |

> **CRL vs. OCSP trap:** CRL = a *downloaded list* (batch, can be stale/large). OCSP = a *real-time single-cert check*. You saw cert validation indirectly in TLS during your HTB/web work.

## 7. Why It Exists

Cryptography exists to make security possible over **untrusted channels and storage**. Without it: anyone on the network reads your traffic (no confidentiality), data can be silently altered (no integrity), and you can't prove who did what (no non-repudiation). PKI/certificates exist to solve **trust at scale** — how do you trust a public key from a stranger's website? Because a CA you already trust vouched for it. Hashing/salting/stretching exist because storing passwords in plaintext is catastrophic, and naive hashing is crackable.

---

## 8. How It Works (enterprise)

- **TLS** (every HTTPS site) uses certificates + asymmetric key exchange to set up a symmetric session key (hybrid model).
- **Full-disk encryption** (BitLocker/LUKS) uses a **TPM** to protect the disk key.
- **Enterprises and CAs** use **HSMs** to generate/store keys securely at scale.
- **Cloud** uses **KMS** (e.g., AWS KMS, Azure Key Vault) to manage keys for services.
- **Password storage** uses **salted + key-stretched** hashes (bcrypt/Argon2).
- **Card data** uses **tokenization** to stay out of PCI scope.
- **Code/documents** use **digital signatures** for integrity and authenticity.

---

## 9. Advantages

- Confidentiality, integrity, authentication, and non-repudiation — the cryptographic foundation of trust.
- Hybrid encryption gives both security (key distribution) and speed.
- Certificates enable scalable trust across the internet.
- Hashing+salting+stretching make password databases far harder to crack.
- Tokenization removes sensitive data from systems entirely (reduces compliance scope).

---

## 10. Disadvantages

- **Key management is hard** — losing a key can mean losing data; protecting keys is critical (hence TPM/HSM/KMS/escrow).
- **Performance cost** — asymmetric is slow; heavy crypto can affect availability.
- **Weak/old algorithms** (MD5, SHA-1, DES, RSA-1024) are exploitable — must be retired.
- **Misuse** — using hashing where you need recoverable data, or a self-signed cert publicly, fails the requirement.
- **Escrow risk** — a central key store is a high-value target.
- **Quantum threat (forward-looking):** future quantum computers may break RSA/ECC, motivating post-quantum cryptography.

## 11. Real-World Examples

- **Confidentiality:** HTTPS/TLS on every secure website; VPN tunnels.
- **Integrity:** verifying a downloaded file's SHA-256 — your `hash-checker` tool does exactly this.
- **Non-repudiation:** a digitally signed software update or contract.
- **Passwords:** a site storing bcrypt(salt+password) — your `password-checker` tool reasons about exactly this strength.
- **FDE:** a stolen laptop's data is unreadable because BitLocker+TPM encrypted the disk.
- **Tokenization:** an e-commerce site stores a token instead of the real card number.
- **Certificates:** your browser trusts `example.com` because a third-party CA signed its cert; an internal tool warns because it's self-signed.

---

## 12. Enterprise Usage (SOC relevance)

- You'll investigate **TLS/certificate issues** (expired/self-signed/revoked certs, weak ciphers) and **downgrade attacks**.
- **Hash verification** is core to malware analysis (a file's SHA-256 → check VirusTotal — your `hash-checker`).
- **Suspicious encryption** can indicate **ransomware** (mass file encryption) or exfiltration over TLS.
- You'll recognize **weak crypto** as a finding (MD5/SHA-1 usage, expired certs, missing PFS).
- **Steganography** and **encrypted channels** are evasion techniques you must be aware of.

---

## 13. Security Implications

- **Weak algorithms/short keys** = breakable confidentiality/integrity.
- **Poor key management** = the single biggest crypto failure mode (stolen/leaked private keys break everything).
- **Self-signed certs on public services** train users to ignore warnings (dangerous habit).
- **Unsalted hashes** are crackable with rainbow tables; **fast hashes** (plain SHA-256) for passwords are crackable at scale — use stretching.
- **Expired/revoked certs not checked** (no OCSP/CRL) let attackers use stolen certs.

---

## 14. Attacker Perspective

- **Brute force / dictionary** against weak keys and unsalted/fast-hashed passwords.
- **Rainbow tables** against unsalted hashes (salting defeats this).
- **Collision attacks** against MD5/SHA-1 (forge a file with the same hash).
- **Downgrade attacks** force weaker crypto (e.g., SSL stripping) — defeated by enforcing strong protocols/HSTS.
- **On-path (MITM)** to intercept key exchange — defeated by certificate validation + PFS.
- **Stealing private keys** (from memory, disk, or a weak KMS) to impersonate or decrypt.
- **Ransomware** weaponizes encryption *against* you.
- Attackers **abuse steganography** and encrypted tunnels to hide payloads/exfiltration.

---

## 15. Defender Perspective

- Use **strong, current algorithms** (AES-256, SHA-256+/SHA-3, RSA-2048+/ECC) and retire weak ones.
- **Salt + key-stretch** all stored passwords (bcrypt/Argon2).
- Protect keys with **TPM/HSM/KMS**; rotate and revoke properly.
- **Validate certificates** (OCSP/CRL); monitor for expired/self-signed/weak certs.
- **Enforce TLS** with PFS; disable downgrade.
- Watch for **mass-encryption** (ransomware) and anomalous encrypted exfiltration.

---

## 16. Detection Techniques

- **Hash-based detection:** match file hashes against threat-intel (your `hash-checker` + VirusTotal); FIM uses hashes to detect tampering.
- **Certificate monitoring:** alert on expired, self-signed, revoked, or weak-cipher certs; certificate transparency logs.
- **Weak-crypto detection:** scanners flag MD5/SHA-1, SSLv3/TLS1.0, short keys.
- **Ransomware indicators:** rapid mass file modification/encryption, extension changes, shadow-copy deletion.
- **Anomalous encryption:** unexpected encrypted channels / large TLS exfiltration.

---

## 17. Mitigation Techniques

- Strong algorithms + adequate key length; deprecate weak ones.
- Robust **key management** (TPM/HSM/KMS, rotation, escrow with strict controls).
- **Salting + key stretching** for passwords.
- **Certificate hygiene** (valid CA, monitor expiry, OCSP/CRL checking, OCSP stapling).
- **Enforce TLS + PFS**; block downgrades (HSTS).
- **Tokenization/masking** to remove sensitive data from systems.
- **Full-disk encryption** for data at rest on endpoints.

---

## 18. Best Practices

1. **Encrypt data at rest and in transit** (FDE + TLS).
2. **Use the hybrid model** — asymmetric for key exchange, symmetric for bulk.
3. **Encrypt with the recipient's public key; sign with your private key.**
4. **Hash for integrity; never "encrypt" when you need irreversibility, and never hash when you need recoverability.**
5. **Always salt + stretch** stored passwords (bcrypt/scrypt/Argon2).
6. **Retire MD5, SHA-1, DES, RSA-1024**; use AES-256, SHA-256+, RSA-2048+/ECC.
7. **Protect private keys in hardware** (TPM/HSM) and manage lifecycle (KMS).
8. **Use third-party certs for public services**; self-signed only internally.
9. **Check revocation** (OCSP preferred; OCSP stapling for efficiency).
10. **Enable PFS** so past traffic stays safe if a key leaks.

## 20. ASCII Diagrams

**Hybrid encryption (TLS-style):**

```
   Client                                  Server
     │  1. Get server cert (public key) ◄──────┐
     │                                          │ (cert signed by CA)
     │  2. Verify cert via CA / OCSP            │
     │  3. Exchange/derive a SYMMETRIC          │
     │     session key (asymmetric/DH)  ───────►│
     │  4. Encrypt bulk data with the           │
     │     fast SYMMETRIC key  ◄──────────────► │
```

**Digital signature:**

```
   SIGN:   message ──hash──► digest ──encrypt w/ PRIVATE key──► signature
   VERIFY: signature ──decrypt w/ PUBLIC key──► digest A
           message ──hash──► digest B
           A == B ?  → integrity + authenticity + non-repudiation
```

**Certificate chain (root of trust):**

```
   Root CA (root of trust)  ──signs──►  Intermediate CA  ──signs──►  Server cert
   (trusted by OS/browser)                                          (example.com)
```

## 21. Exam Tips

- **Encrypt with the recipient's PUBLIC key; sign with your PRIVATE key.** This single rule answers many questions.
- **Hashing = integrity (one-way). Encryption = confidentiality (reversible).** Don't swap.
- **Digital signature = integrity + authentication + non-repudiation** (but NOT confidentiality by itself).
- **Symmetric = fast/bulk; Asymmetric = key exchange + signatures.** Real systems use **both (hybrid)**.
- **TPM = one device; HSM = enterprise appliance.** KMS = manages key lifecycle.
- **Tokenization** replaces data with a token (vault lookup); **masking** shows partial data; **steganography** hides data inside other data — none are "encryption."
- **Salting defeats rainbow tables; key stretching (bcrypt/PBKDF2/Argon2) slows brute force.**
- **CRL = list (batch); OCSP = real-time single check.**
- **Self-signed = internal/testing only; third-party CA = public trust.**
- **CSR contains the public key + identity — never the private key.**
- **ECC = strong with shorter keys** (efficient for mobile/IoT).
- **MD5/SHA-1 = broken (collisions); DES/3DES, RSA-1024 = weak/legacy.**

## 22. Common CompTIA Question Styles

1. **Requirement→tool:** *"Which provides integrity verification of a downloaded file?"* → Hashing (SHA-256).
2. **Which key:** *"To send a confidential message to Bob, which key do you use?"* → Bob's public key.
3. **Signature:** *"Which provides non-repudiation?"* → Digital signature (sign with private key).
4. **Tokenization vs. encryption:** *"Replace card numbers with values that have no mathematical relationship to the original?"* → Tokenization.
5. **TPM vs. HSM:** *"A chip on the motherboard storing the disk-encryption key?"* → TPM.
6. **CRL vs. OCSP:** *"Real-time check of a single certificate's revocation status?"* → OCSP.
7. **Cert type:** *"A cert for `app.example.com`, `mail.example.com`, etc. with one cert?"* → Wildcard.
8. **Password storage:** *"Defeats rainbow tables?"* → Salting. *"Makes brute force slow/expensive?"* → Key stretching.
9. **Weak crypto:** *"Which hashing algorithm should be replaced due to collision attacks?"* → MD5 (or SHA-1).

**Why distractors fail:** they swap public/private keys, hashing/encryption, CRL/OCSP, TPM/HSM, or tokenization/masking. Lock onto the exact requirement verb.

## 23. Interview Notes

- *"Symmetric vs. asymmetric — when use each?"* — Symmetric for bulk (fast); asymmetric for key exchange + signatures; explain the hybrid model (TLS).
- *"How are passwords stored securely?"* — Salted + key-stretched hash (bcrypt/Argon2), never plaintext or fast unsalted hashes.
- *"Walk me through a TLS handshake."* — Cert validation → asymmetric key exchange → symmetric session key → encrypted data.
- *"CRL vs. OCSP?"* — List vs. real-time check; mention OCSP stapling.
- *"What's the difference between hashing and encryption?"* — One-way integrity vs. reversible confidentiality.
- *"How would you spot weak crypto in an environment?"* — Scanners flagging MD5/SHA-1, old TLS, short keys, expired/self-signed certs.

---

## 24. Common Beginner Mistakes

1. **Swapping hashing and encryption** (one-way vs. reversible).
2. **Using the wrong key** (signing with public, or encrypting with private).
3. Thinking a **digital signature hides the message** (it doesn't — add encryption for confidentiality).
4. Confusing **TPM (device) with HSM (enterprise appliance)**.
5. Thinking **tokenization = encryption** (it's a vault swap, not a key transform).
6. Forgetting **salt** (rainbow-table-vulnerable) or using a **fast hash** for passwords (no stretching).
7. Using **self-signed certs** on public services.
8. Believing **MD5/SHA-1** are still safe.
9. Confusing **CRL (list)** with **OCSP (real-time)**.
10. Putting the **private key in a CSR** (a CSR carries only the public key + identity).

## 25. Related Concepts

## 26. Memory Tricks

- **"Encrypt to the recipient's PUBLIC; sign with your PRIVATE."** (The master rule.)
- **Hashing = "shred"** — one-way, can't unshred. **Encryption = "lock"** — open it with the key.
- **Signature = "Hash + Private-key seal"** → integrity + identity + non-repudiation.
- **Hybrid = "asymmetric carries the symmetric key"** (slow car delivers the fast key).
- **TPM = "T for The device" (one); HSM = "H for Huge/enterprise."**
- **Salt then Stretch** → *"Season it, then slow it"* (salt defeats rainbow tables; stretch slows cracking).
- **CRL = "the Roll call List" (batch); OCSP = "On-call, real-time."**
- **Tokenization = swap for a fake; Masking = hide most of it; Steganography = hide it inside something else.**
- **CSR carries the PUBLIC key only** — *"you never mail your private key."*

## 27. Key Facts

- **Encryption = confidentiality (reversible); Hashing = integrity (one-way); Signatures = integrity + auth + non-repudiation.**
- **Symmetric (AES) = fast/bulk; Asymmetric (RSA/ECC/DH) = key exchange + signatures; real systems are hybrid.**
- **Encrypt to recipient's public key; decrypt with your private key; sign with your private; verify with sender's public.**
- **PKI** manages keys/trust via **CAs** and **certificates**; **key escrow** = recoverable key backup.
- **Encryption levels:** full-disk, partition, volume, file, database, record; plus transport (TLS).
- **Tools:** TPM (device), HSM (enterprise), KMS (lifecycle), secure enclave (CPU).
- **Obfuscation ≠ encryption:** steganography (hide inside), tokenization (vault swap), masking (partial).
- **Salting** defeats rainbow tables; **key stretching** (bcrypt/PBKDF2/Argon2) slows brute force.
- **Certificates:** CSR (public key+identity) → CA signs; revocation via **CRL (list)** or **OCSP (real-time)**; **self-signed = internal**, **third-party = public**; **wildcard = all subdomains**; **root of trust = root CA**.
- **Weak/deprecated:** MD5, SHA-1, DES, RSA-1024. **Strong:** AES-256, SHA-256+/SHA-3, RSA-2048+/ECC.
- **Blockchain = distributed, append-only, tamper-evident ledger; open public ledger = transparent/decentralized.**

## 28. Quick Revision Sheet

GOAL → TOOL:
  Confidentiality → Encryption     Integrity → Hashing
  Non-repudiation → Digital signature (sign w/ PRIVATE)
  Key distribution → Asymmetric key exchange (then symmetric for bulk = HYBRID)

  Encrypt → recipient's PUBLIC | Decrypt → your PRIVATE
  Sign    → your PRIVATE       | Verify  → sender's PUBLIC

SYMMETRIC: AES(128/192/256), 3DES(legacy), ChaCha20   (fast, shared key)
ASYMMETRIC: RSA, ECC, DH                              (slow, key pair)
ECC = strong with SHORT keys (mobile/IoT)

ENCRYPTION LEVELS: full-disk | partition | volume | file | database | record | transport(TLS)

TOOLS: TPM(device) | HSM(enterprise) | KMS(lifecycle) | Secure enclave(CPU)

OBFUSCATION (not encryption): steganography(hide inside) | tokenization(vault swap) | masking(partial)

HASHING: one-way, integrity. Good=SHA-256/3; BROKEN=MD5,SHA-1
  + SALT (defeat rainbow tables) + KEY STRETCH (bcrypt/PBKDF2/Argon2 = slow brute force)

  CSR = public key + identity (NEVER private key) → CA signs

## 29. Practice Questions

**Q1.** To send Alice a confidential file, which key should Bob use to encrypt it?
A. Bob's private key  B. Bob's public key  C. Alice's public key  D. Alice's private key

**Q2.** Which provides integrity verification but cannot be reversed to recover the original data?
A. AES encryption  B. RSA encryption  C. Hashing  D. Tokenization

**Q3.** A developer wants to store user passwords securely and defeat rainbow-table attacks. What should be added before hashing?
A. A salt  B. A wildcard  C. A CSR  D. An OCSP responder

**Q4.** Which device is a tamper-resistant appliance used by enterprises to generate and store cryptographic keys at scale?
A. TPM  B. HSM  C. Secure enclave  D. KMS

**Q5.** Which provides a real-time check of a single certificate's revocation status?
A. CRL  B. CSR  C. OCSP  D. CA

**Q6.** A company replaces credit card numbers with surrogate values that have no mathematical relationship to the originals, storing the real values in a secure vault. This is:
A. Encryption  B. Hashing  C. Tokenization  D. Steganography

**Q7.** Which provides non-repudiation?
A. Symmetric encryption  B. Digital signature  C. Salting  D. Full-disk encryption

**Q8.** Which hashing algorithm should be deprecated because practical collision attacks exist?
A. SHA-256  B. SHA-3  C. MD5  D. bcrypt

**Q9.** Which certificate would secure `mail.example.com`, `app.example.com`, and `vpn.example.com` with a single certificate?
A. Self-signed certificate  B. Wildcard certificate  C. OCSP certificate  D. Root certificate

**Q10.** Which technique makes brute-forcing hashed passwords computationally expensive by performing many iterations?
A. Salting  B. Key stretching  C. Steganography  D. Masking

**Q11.** In a typical TLS session, why are both asymmetric and symmetric encryption used?
A. Asymmetric is faster for bulk data
B. Symmetric distributes keys securely
C. Asymmetric securely exchanges a symmetric key, then symmetric encrypts the data
D. Both are required by law

**Q12.** What does a Certificate Signing Request (CSR) contain?
A. The private key  B. The public key and identity information  C. The CRL  D. The session key

### Answer Key & Explanations

- **Q1 — C (Alice's public key).** Encrypt to the recipient with *their* public key; only Alice's private key can decrypt.
- **Q2 — C (Hashing).** One-way, integrity, not reversible. A/B are reversible encryption; D is a vault swap.
- **Q3 — A (Salt).** Random salt makes identical passwords hash differently, defeating rainbow tables.
- **Q4 — B (HSM).** Enterprise tamper-resistant key appliance. TPM is per-device; KMS manages lifecycle (software/service); secure enclave is a CPU region.
- **Q5 — C (OCSP).** Real-time single-cert status. CRL is a downloaded batch list.
- **Q6 — C (Tokenization).** Surrogate token + secure vault, no mathematical relationship.
- **Q7 — B (Digital signature).** Signing with the private key proves origin → non-repudiation.
- **Q8 — C (MD5).** Practical collisions exist (also SHA-1). SHA-256/SHA-3 are strong; bcrypt is for password stretching.
- **Q9 — B (Wildcard).** `*.example.com` covers all those subdomains.
- **Q10 — B (Key stretching).** Many iterations (bcrypt/PBKDF2/Argon2) slow each brute-force guess.
- **Q11 — C.** Hybrid model: asymmetric exchanges the symmetric key; symmetric (fast) encrypts the bulk traffic.
- **Q12 — B.** A CSR carries the public key + identity; the private key never leaves you.

## 30. PBQ-style Scenario

> **Scenario:** You're securing a new web application and its data. For each requirement, choose the appropriate cryptographic solution.

| # | Requirement | Best solution |
|---|-------------|---------------|
| 1 | Make the website's traffic confidential in transit | ? |
| 2 | Prove a downloaded installer hasn't been tampered with | ? |
| 3 | Store user passwords so a database leak doesn't expose them and brute force is slow | ? |
| 4 | Protect the disk-encryption key on each employee laptop | ? |
| 5 | Get a publicly trusted certificate for `www.example.com` | ? |
| 6 | Check in real time whether the site's certificate has been revoked | ? |
| 7 | Keep credit card numbers out of PCI scope by replacing them with non-reversible surrogates | ? |
| 8 | Let the sender of a contract be unable to deny they signed it | ? |

**Answer Key:**

1. **TLS encryption** (transport/communication encryption; hybrid asymmetric+symmetric).
2. **Hashing** (publish/compare a SHA-256 digest) — optionally a **digital signature** for authenticity too.
3. **Salted + key-stretched hashing** (bcrypt/Argon2).
4. **TPM** (per-device chip protecting the FDE key).
5. **Third-party CA certificate** (generate a key pair → **CSR** → public CA signs).
6. **OCSP** (real-time revocation check; CRL is the batch alternative).
7. **Tokenization** (surrogate tokens + secure vault).
8. **Digital signature** (sign with the sender's private key → non-repudiation).
