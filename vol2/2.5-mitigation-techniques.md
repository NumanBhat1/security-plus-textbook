## 4. Concept Overview

After learning *what can go wrong* (vectors, vulnerabilities, attacks), this objective covers *what to do about it* — the practical controls that reduce risk across the enterprise. It's the "given this threat, pick the mitigation" objective. Many of these reappear throughout the cert, so mastering them here pays off broadly.

The techniques cluster naturally:

- **Limit reach:** segmentation, isolation, least privilege, access control (ACLs/permissions).
- **Control software:** application allow list, patching, removal of unnecessary software, decommissioning.
- **Protect data:** encryption.
- **Maintain a secure baseline:** configuration enforcement, hardening (endpoint protection, host firewall, HIPS, disable ports/protocols, change default passwords).
- **Watch:** monitoring.

## 5. Definitions

| Term | Definition |
|------|------------|
| **Segmentation** | Dividing a network into separate zones to limit lateral movement. |
| **Access control** | Restricting who/what can access resources. |
| **Access control list (ACL)** | Rules specifying allowed/denied access (by IP, user, port). |
| **Permissions** | Rights granted to users/groups on resources (read/write/execute). |

| **Least privilege** | Granting only the minimum access needed. |
| **Configuration enforcement** | Ensuring systems stay at a secure, approved baseline. |
| **Decommissioning** | Securely retiring and removing systems/data. |
| **Hardening** | Reducing a system's attack surface to a secure baseline. |

| **Host-based firewall** | A firewall running on an individual host. |
| **HIPS** | Host-based intrusion prevention system — blocks malicious host activity. |

## 6. Deep Technical Explanation

### Limiting reach

- **Segmentation** — split the network into zones (VLANs, subnets, DMZ) so a breach in one area **can't move laterally** to others. Limits blast radius (and contains worms — 2.4).
- **Isolation** — go further: **fully separate** a system/process (air-gap, sandbox, quarantine an infected host). Segmentation *divides into zones*; isolation *completely cuts off*.
- **Least privilege** — users/processes get **only the access they need**, nothing more. Limits what a compromised account/process can do (also blunts privilege escalation and insider misuse).
- **Access control (ACLs & permissions):**
  - **ACL** — rule lists that **allow/deny** traffic or access (e.g., a firewall/router ACL by IP/port, or a filesystem ACL).
  - **Permissions** — the specific **rights** (read/write/execute) granted to users/groups on objects.

> **Segmentation vs. isolation:** segmentation **divides** the network into communicating zones; isolation **completely separates** (no/minimal connection). **ACL vs. permissions:** ACLs are *rules controlling access* (often network/object-level); permissions are the *granted rights* on a resource.

### Controlling software

- **Application allow list** (allowlisting) — **only approved apps run**; everything else is blocked by default. Strong against malware/unapproved software (opposite of a block/deny list, which blocks only known-bad).
- **Patching** — fix known vulnerabilities promptly (the single highest-value hygiene control).
- **Removal of unnecessary software** — fewer apps = smaller attack surface.
- **Decommissioning** — securely retire end-of-life systems: remove from the network, **sanitize data** (wipe/destroy), revoke accounts/certs. Eliminates the unpatchable-EOL risk (2.3).

### Protecting data

- **Encryption** — protect data **at rest, in transit, and in use** so stolen data is unreadable. Defends confidentiality even after a breach (details in 1.4).

### Maintaining a secure baseline

- **Configuration enforcement** — continuously ensure systems match the **approved secure baseline** (and auto-remediate drift). Stops misconfigurations (2.3) and unauthorized changes.
- **Hardening techniques** — reduce the attack surface to a secure baseline:

| Hardening technique | Purpose |
|---------------------|---------|
| **Encryption** | Protect data on the host |
| **Endpoint protection** | Anti-malware/EDR detects & blocks malware |
| **Host-based firewall** | Filter traffic to/from the individual host |
| **HIPS** | Detect & **prevent** malicious activity on the host |
| **Disabling ports/protocols** | Close unused services (shrinks attack surface) |
| **Default password changes** | Eliminate the default-credentials vector (2.2) |
| **Removal of unnecessary software** | Fewer programs = fewer vulnerabilities |

### Watching

- **Monitoring** — continuous observation (SIEM, EDR, logs) to **detect** what prevention misses. Mitigation isn't only prevention — detection enables fast response (2.4 indicators).

## 7. Why It Exists

You can't eliminate all risk, so you **layer mitigations** (defense in depth) to prevent, limit, and detect. This objective ties the whole domain together: each threat/vulnerability/attack from 2.1–2.4 has a corresponding control here. CompTIA wants you to pick the *right* mitigation for a given scenario — the everyday decision of a security practitioner.

---

## 8. How It Works (enterprise)

Organizations apply these in layers: **segmentation** structures the network; **least privilege + access control** govern who reaches what; **allow lists, patching, configuration enforcement, and hardening** keep endpoints/servers secure; **encryption** protects data; **monitoring** watches everything; and **decommissioning** retires risk. Baselines (CIS Benchmarks) define hardening; tools (MDM, GPO, config management, CSPM) enforce them; SIEM/EDR provide monitoring.

---

## 9. Advantages

- **Defense in depth** — multiple layers mean one failure isn't fatal.
- **Reduced attack surface** (hardening, removal, disabling ports).
- **Contained breaches** (segmentation/isolation/least privilege).
- **Faster detection/response** (monitoring).
- **Data protected even if stolen** (encryption).

---

## 10. Disadvantages / Challenges

- **Operational overhead** — patching, allow lists, and config enforcement take effort.
- **Usability friction** — strict least privilege/allow lists can block legitimate work.
- **Complexity** — over-segmentation is hard to manage.
- **Coverage gaps** — monitoring without tuning = alert fatigue.
- **Key management** — encryption is only as good as its key handling.

---

## 11. Real-World Examples

- **Segmentation:** isolating the payment network (PCI) from corporate LAN.
- **Isolation:** quarantining an infected host off the network during IR.
- **Application allow list:** only signed, approved binaries run on POS terminals.
- **Patching:** monthly patch cycle closing known CVEs.
- **Least privilege:** standard users can't install software.
- **Default password changes + disabling ports:** hardening a new server.
- **Decommissioning:** wiping and retiring EOL servers.
- **Encryption:** full-disk encryption on laptops.

---

## 12. Enterprise Usage (SOC relevance)

- **Monitoring** is your domain — SIEM/EDR feed your alerts.
- **Segmentation/isolation** are containment actions you recommend/execute during IR.
- **Least privilege & allow lists** reduce the incidents you handle.
- **Patching gaps and missing hardening** show up as recurring findings you report.
- When you identify an attack (2.4), you map it to the **mitigation** here (e.g., worm → segmentation + patching; default creds → change them + disable ports).

---

## 13. Security Implications

- **No segmentation** = flat network where one breach spreads everywhere.
- **Over-privilege** = a single compromise becomes domain-wide.
- **Unpatched/unhardened** systems are the most exploited.
- **No monitoring** = breaches go undetected for months.
- **Improper decommissioning** = data left on retired/disposed assets.

---

## 14. Attacker / Adversary Perspective

- Attackers love **flat networks** (no segmentation) for easy lateral movement.
- They seek **over-privileged accounts** to escalate quickly.
- They target **unpatched, unhardened** systems with default creds and open ports.
- They try to **evade monitoring** (clear logs, live off the land).
- **Allow lists and least privilege** are the controls that most frustrate them.

---

## 15. Defender Perspective

- **Layer everything** (defense in depth) — assume any one control can fail.
- **Segment + least privilege** to shrink blast radius.
- **Patch and harden** to a baseline; enforce config continuously.
- **Allow list** critical/high-risk endpoints.
- **Encrypt** sensitive data everywhere.
- **Monitor** comprehensively and tune for signal.
- **Decommission** EOL assets securely.

---

## 16. Detection / Enforcement Techniques

- **SIEM/EDR/IDS** for monitoring.
- **Configuration management / CSPM / GPO** for configuration enforcement and drift detection.
- **HIPS / host firewall / endpoint protection** for host-level prevention.
- **NAC + segmentation** to enforce network zones.
- **Allow-list enforcement** (e.g., WDAC/AppLocker) blocking unapproved binaries.

## 18. Best Practices

1. **Apply defense in depth** — layer these controls.
2. **Segment networks; isolate** during incidents.
3. **Enforce least privilege** and tight access control (ACLs + permissions).
4. **Patch promptly; harden to a baseline (CIS); enforce config.**
5. **Allow-list** high-risk/critical systems.
6. **Encrypt** data at rest and in transit.
7. **Monitor** comprehensively; tune alerts.
8. **Decommission EOL** systems securely (sanitize data).

## 20. Diagrams

**Segmentation vs. isolation:**

```
   SEGMENTATION:  [ Zone A ] ─allowed rules─ [ Zone B ] ─rules─ [ DMZ ]
                  (divided, but zones still communicate via controlled rules)

   ISOLATION:     [ Quarantined host ]   ✂   (no connection to anything)
```

**Allow list vs. block list:**

```
   ALLOW LIST:  default = DENY → only approved apps run   (strong)
   BLOCK LIST:  default = ALLOW → only known-bad blocked   (weaker)
```

**Defense in depth:**

```
   Network: segmentation ─► Access: least privilege/ACL ─► Host: hardening/HIPS/EDR
   ─► Data: encryption ─► Everywhere: MONITORING
```

## 21. Exam Tips

- **Segmentation = divide into zones; isolation = fully separate/cut off.**
- **ACL = access *rules*; permissions = granted *rights* (read/write/execute).**
- **Application allow list = default-deny (only approved runs)** — strongest against unknown malware.
- **Least privilege = minimum access** (limits blast radius and privilege escalation).
- **Configuration enforcement = keep systems at the secure baseline** (stops drift/misconfig).
- **Decommissioning = securely retire EOL systems** (sanitize data).
- **Hardening list:** encryption, endpoint protection, host firewall, HIPS, disable ports/protocols, change default passwords, remove unnecessary software.
- **HIPS prevents (blocks); monitoring detects.**
- **Patching** is the highest-value hygiene control.
- For "given this threat, what mitigation?" — match using the **threat→control table** above.

## 22. Common CompTIA Question Styles

1. *"How do you stop a worm from spreading laterally across the network?"* → Segmentation (+ patching, isolation).
2. *"Only pre-approved applications should be allowed to run on kiosks."* → Application allow list.
3. *"Users should not have more access than their job requires."* → Least privilege.
4. *"A new server still uses admin/admin and has unused services running."* → Change default passwords + disable ports/protocols (hardening).
5. *"Ensure systems don't drift from the approved secure baseline."* → Configuration enforcement.
6. *"Securely retire and wipe an end-of-life server."* → Decommissioning.
7. *"What completely separates an infected host from the network?"* → Isolation.
8. *"Which mitigation detects malicious activity rather than preventing it?"* → Monitoring.

**Why distractors fail:** they swap segmentation/isolation, ACL/permissions, allow/block list, or HIPS (prevent) vs. monitoring (detect). Anchor on *divide vs. separate*, *rules vs. rights*, *default-deny vs. default-allow*, *prevent vs. detect*.

## 23. Interview Notes

- *"Segmentation vs. isolation?"* — Segmentation divides the network into controlled zones; isolation fully separates a system (e.g., quarantine).
- *"What's an application allow list?"* — Default-deny: only explicitly approved apps run; everything else is blocked.
- *"Why least privilege?"* — It limits the damage a compromised account or process can do.
- *"What is configuration enforcement?"* — Keeping systems at an approved secure baseline and remediating drift.
- *"Name key hardening steps."* — Patch, disable unused ports/services, change default passwords, install EDR/host firewall/HIPS, remove unneeded software, encrypt.

## 24. Common Beginner Mistakes

1. **Confusing segmentation and isolation.**
2. **Mixing ACLs (rules) with permissions (rights).**
3. **Thinking a block list is as strong as an allow list** (it isn't — allow list is default-deny).
4. **Forgetting decommissioning includes data sanitization.**
5. **Treating HIPS and monitoring as the same** (prevent vs. detect).
6. **Overlooking "change default passwords" as a hardening step.**
7. **Assuming patching alone is enough** — layering is required.

## 27. Key Facts

- **Limit reach:** segmentation (zones), isolation (fully separate), least privilege (minimum access), access control (ACLs = rules, permissions = rights).
- **Control software:** application allow list (default-deny), patching, remove unnecessary software, decommissioning (retire + sanitize).
- **Protect data:** encryption (rest/transit/use).
- **Baseline:** configuration enforcement; hardening = encryption + endpoint protection + host firewall + HIPS + disable ports/protocols + change default passwords + remove unneeded software.
- **Detect:** monitoring (HIPS prevents; monitoring detects).
- **Allow list (default-deny) > block list (default-allow).**

## 29. Practice Questions

**Q1.** To prevent a compromised host in one part of the network from reaching critical servers, you divide the network into controlled zones. This is:
A. Isolation  B. Segmentation  C. Encryption  D. Monitoring

**Q2.** A security tool allows ONLY explicitly approved applications to run and blocks everything else by default. This is a(n):
A. Block/deny list  B. Application allow list  C. ACL  D. HIPS

**Q3.** Users are granted only the minimum access required to perform their jobs. This principle is:
A. Separation of duties  B. Least privilege  C. Configuration enforcement  D. Segmentation

**Q4.** During an incident, you completely disconnect an infected machine from the network. This is:
A. Segmentation  B. Decommissioning  C. Isolation  D. Patching

**Q5.** Which best describes the difference between segmentation and isolation?
A. They are identical
B. Segmentation divides the network into communicating zones; isolation fully separates a system
C. Isolation divides into zones; segmentation fully separates
D. Both mean encrypting traffic

**Q6.** Ensuring all systems continuously match the approved secure baseline and remediating any drift is:
A. Monitoring  B. Configuration enforcement  C. Decommissioning  D. Encryption

**Q7.** Which is NOT a hardening technique listed for SY0-701?
A. Disabling unused ports/protocols
B. Changing default passwords
C. Increasing user privileges for convenience
D. Installing endpoint protection

**Q8.** Securely retiring an end-of-life server, including wiping its data and revoking its accounts, is:
A. Isolation  B. Decommissioning  C. Segmentation  D. Patching

**Q9.** Which control DETECTS malicious activity rather than preventing it?
A. HIPS  B. Application allow list  C. Monitoring  D. Host-based firewall

**Q10.** A firewall rule that permits or denies traffic based on IP address and port is an example of a(n):
A. Permission  B. Access control list (ACL)  C. Allow list  D. Logic bomb

### Answer Key & Explanations

- **Q1 — B (Segmentation).** Dividing into zones limits lateral movement.
- **Q2 — B (Application allow list).** Default-deny; only approved apps run.
- **Q3 — B (Least privilege).** Minimum necessary access.
- **Q4 — C (Isolation).** Fully separating the host.
- **Q5 — B.** Segmentation = zones; isolation = full separation.
- **Q6 — B (Configuration enforcement).** Keeps systems on the baseline.
- **Q7 — C.** Increasing privileges is the opposite of hardening.
- **Q8 — B (Decommissioning).** Secure retirement + data sanitization.
- **Q9 — C (Monitoring).** Detects (HIPS/firewall/allow list prevent).
- **Q10 — B (ACL).** Rules allowing/denying by IP/port.
