> - **Physical attacks** — Brute force; Radio frequency identification (RFID) cloning; Environmental
> - **Network attacks** — Distributed denial-of-service (DDoS) (Amplified, Reflected); Domain Name System (DNS) attacks; Wireless; On-path; Credential replay; Malicious code
> - **Application attacks** — Injection; Buffer overflow; Replay; Privilege escalation; Forgery; Directory traversal
> - **Cryptographic attacks** — Downgrade; Collision; Birthday
> - **Password attacks** — Spraying; Brute force
> - **Indicators** — Account lockout; Concurrent session usage; Blocked content; Impossible travel; Resource consumption; Resource inaccessibility; Out-of-cycle logging; Published/documented; Missing logs

---

## 3. Learning Outcomes

After this objective you will be able to:

1. Identify all nine **malware types** and the indicator that distinguishes each.
2. Recognize **physical, network, application, cryptographic,** and **password** attacks from a scenario.
3. Interpret the official **indicators of malicious activity** (impossible travel, missing logs, resource consumption, etc.).
4. Map an indicator/scenario to the correct attack — the core skill this objective tests.

---

## 4. Concept Overview

This is the **largest and most scenario-heavy objective in Domain 2** — it's the "name that attack" objective. CompTIA gives you a behavior or symptom (an *indicator*) and asks which attack it signals. So success here is about crisp definitions and the **one distinguishing detail** for each attack and malware type.

Six attack families plus a list of indicators:

1. **Malware** — 9 types (ransomware, trojan, worm, spyware, bloatware, virus, keylogger, logic bomb, rootkit).
2. **Physical** — brute force, RFID cloning, environmental.
3. **Network** — DDoS (amplified/reflected), DNS attacks, wireless, on-path, credential replay, malicious code.
4. **Application** — injection, buffer overflow, replay, privilege escalation, forgery, directory traversal.
5. **Cryptographic** — downgrade, collision, birthday.
6. **Password** — spraying, brute force.

Then the **indicators** — the observable signs (impossible travel, account lockout, missing logs…) that tell you something malicious is happening.

## 5. Definitions

| Term | Definition |
|------|------------|
| **Malware** | Malicious software designed to harm or exploit. |
| **Ransomware** | Encrypts data and demands payment for the key. |
| **Trojan** | Malware disguised as legitimate software. |
| **Worm** | Self-replicating malware that spreads **without** user action. |
| **Spyware** | Secretly gathers user info/activity. |
| **Bloatware** | Unwanted pre-installed software (not always malicious, but risky/unneeded). |
| **Virus** | Malware that attaches to a file/program and **needs the user to run it** to spread. |
| **Keylogger** | Records keystrokes to steal credentials/data. |
| **Logic bomb** | Malicious code that triggers on a condition/date. |
| **Rootkit** | Malware that hides deep in the system (often kernel) to evade detection. |
| **RFID cloning** | Copying an RFID badge/card to impersonate it. |
| **DDoS** | Overwhelming a target with traffic from many sources. |
| **Amplified DDoS** | Small requests trigger large responses to the victim. |
| **Reflected DDoS** | Traffic bounced off third parties (spoofed source) to the victim. |
| **On-path attack** | Attacker intercepts/relays traffic between two parties (formerly MITM). |
| **Credential replay** | Reusing captured credentials/tokens to authenticate. |
| **Privilege escalation** | Gaining higher permissions than authorized. |
| **Directory traversal** | Accessing files outside the intended directory (`../`). |
| **Forgery** | Faking data/requests (e.g., CSRF/SSRF, request forgery). |
| **Downgrade attack** | Forcing use of a weaker protocol/cipher. |
| **Collision** | Two different inputs producing the same hash. |
| **Birthday attack** | Exploiting hash collision probability (birthday paradox). |
| **Password spraying** | Trying a few common passwords across **many** accounts. |
| **Brute force** | Trying many passwords against an account (or keys against a cipher). |
| **Impossible travel** | Logins from distant locations too close in time to be real. |

## 6. Deep Technical Explanation

### Malware (the 9 types — know the distinguishing trait)

| Malware | Distinguishing trait |
|---------|----------------------|
| **Virus** | Attaches to a file/program; **needs the user to execute it** to spread |
| **Worm** | **Self-replicates and spreads on its own** (no user action) — network propagation |
| **Trojan** | **Disguised** as legitimate software (no self-replication) |
| **Ransomware** | **Encrypts** data, demands payment |
| **Spyware** | **Secretly monitors** the user |
| **Keylogger** | **Records keystrokes** |
| **Rootkit** | **Hides deep** (kernel) to evade detection; hard to remove |
| **Logic bomb** | **Triggers on a condition** (date/event) |
| **Bloatware** | **Unwanted pre-installed** software (often not malicious, but unneeded/risky) |

> **The #1 malware trap: virus vs. worm.** A **virus needs a user to run it** (attaches to a file); a **worm spreads by itself** across the network. **Trojan** is the one defined by *disguise*.

> **Rootkit** is the stealth one — it operates at a low level (often kernel) to hide processes/files; standard tools may not see it.

### Physical attacks

| Attack | What it is |
|--------|-----------|
| **Brute force (physical)** | Forcing physical entry (breaking a lock/door) |
| **RFID cloning** | Copying an access badge/card to clone its credential |
| **Environmental** | Attacking via the environment (power, HVAC, fire, cooling) to disrupt |

### Network attacks

- **DDoS** — flood a target from many sources to deny service.
  - **Amplified** — small spoofed requests to a service that returns *large* responses (DNS/NTP amplification), multiplying volume at the victim.
  - **Reflected** — requests with the *victim's spoofed source IP* sent to third parties, which reply to the victim.
- **DNS attacks** — poisoning/spoofing DNS, DNS hijacking, or domain manipulation to redirect users.
- **Wireless** — attacks on Wi-Fi (deauth, evil twin, rogue AP).
- **On-path** (formerly **man-in-the-middle**) — attacker sits between two parties, intercepting/altering traffic.
- **Credential replay** — capturing valid credentials/tokens and **reusing** them to authenticate.
- **Malicious code** — running attacker code on the network/host (scripts, exploits).

### Application attacks

| Attack | What it is |
|--------|-----------|
| **Injection** | Inserting malicious input (SQLi, command injection) |
| **Buffer overflow** | Overwriting adjacent memory (2.3) |
| **Replay** | Re-sending captured legitimate requests/tokens |
| **Privilege escalation** | Gaining higher rights (**vertical** = higher role; **horizontal** = another same-level user) |
| **Forgery** | Faking requests — **CSRF** (trick a user's browser) / **SSRF** (trick the server) |
| **Directory traversal** | Using `../` to read files outside the web root (e.g., `/etc/passwd`) |

### Cryptographic attacks

| Attack | What it is |
|--------|-----------|
| **Downgrade** | Forcing a weaker protocol/cipher (e.g., TLS→SSL) to exploit it |
| **Collision** | Finding two inputs with the **same hash** (breaks integrity) |
| **Birthday** | Using the **birthday paradox** to find hash collisions faster than expected |

> **Collision vs. birthday:** a *collision* is the result (two inputs, one hash); the *birthday attack* is the *technique* that exploits collision probability to find them efficiently.

### Password attacks

| Attack | What it is | Avoids lockout? |
|--------|-----------|-----------------|
| **Brute force** | Try **many passwords** against **one** account | No (triggers lockout) |
| **Spraying** | Try **a few common passwords** across **many** accounts | **Yes** (low attempts per account) |

> **Spraying vs. brute force:** brute force hammers *one account* with many guesses; **spraying** uses *one or few common passwords* across *many accounts* to stay under lockout thresholds. "A few passwords, many accounts" = spraying.

### Indicators of malicious activity (the observable signs)

| Indicator | What it signals |
|-----------|-----------------|
| **Account lockout** | Repeated failed logins (brute force) |
| **Concurrent session usage** | Same account active in two places (compromise) |
| **Blocked content** | Security tools blocking malicious payloads/sites |
| **Impossible travel** | Logins from far-apart locations too quickly (stolen creds) |
| **Resource consumption** | Spikes in CPU/RAM/bandwidth (malware, mining, DDoS) |
| **Resource inaccessibility** | Systems/data unavailable (ransomware, DoS) |
| **Out-of-cycle logging** | Log entries at unusual times (suspicious activity) |
| **Published/documented** | Attack publicly known/reported (e.g., disclosed breach, leaked data) |
| **Missing logs** | Gaps where logs should be (attacker covering tracks) |

> **Missing logs** is a classic exam answer for "attacker hiding evidence." **Impossible travel** = stolen credentials. **Resource inaccessibility** = ransomware/DoS. **Account lockout** = brute force in progress.

## 7. Why It Exists

This objective exists because **detection** is the heart of security operations — a SOC analyst's job is reading indicators and naming the attack. CompTIA frames it as "given a scenario, analyze indicators," mirroring real triage: you rarely see the attacker, you see the *symptoms*, and you must infer what's happening to respond correctly.

---

## 8. How It Works (enterprise)

SIEMs, EDR, IDS/IPS, and DLP generate **indicators**; analysts correlate them to identify the attack. UEBA flags impossible travel and concurrent sessions; resource monitoring catches consumption spikes; log integrity monitoring catches missing/out-of-cycle logs. Each attack maps to detections (failed-login thresholds → brute force; large outbound spikes → exfil/DDoS) and to a response playbook (isolate, reset creds, block, restore).

---

## 9. Advantages (of indicator analysis)

- **Early detection** — symptoms surface attacks you can't see directly.
- **Correct response** — naming the attack drives the right containment.
- **Correlation power** — combining indicators reduces false positives.
- **Threat hunting** — indicators guide proactive searches.

---

## 10. Disadvantages / Challenges

- **False positives** — legitimate activity can mimic indicators (VPN ≈ impossible travel).
- **Stealth** — rootkits and skilled actors suppress indicators (missing logs).
- **Volume** — alert fatigue from too many indicators.
- **Encrypted/obfuscated** malware evades signatures.

---

## 11. Real-World Examples

- **Ransomware:** files encrypted, ransom note, resource inaccessibility.
- **Worm:** rapid self-spread across the network (WannaCry-style).
- **Impossible travel:** login from two countries minutes apart → stolen creds.
- **Password spraying:** `Spring2024!` tried across hundreds of accounts.
- **Amplified DDoS:** DNS/NTP reflection flooding a victim.
- **Directory traversal:** `../../etc/passwd` in a URL.
- **Downgrade:** forcing TLS to an old, breakable version.
- **Missing logs:** an attacker clears event logs to hide activity.

---

## 12. Enterprise Usage (SOC relevance)

This objective **is** the SOC job. You'll live in these indicators daily:

- **Brute force** → failed-login spikes / account lockouts in the SIEM.
- **Impossible travel / concurrent sessions** → UEBA alerts (account takeover).
- **Resource consumption** → host alerts (cryptominer, malware).
- **Missing/out-of-cycle logs** → integrity alerts (attacker anti-forensics).
- Recognizing **malware type** (worm vs. trojan) shapes containment (network isolation vs. host cleanup).

---

## 13. Security Implications

- **Worms/ransomware** spread fast — speed of containment matters.
- **Rootkits** undermine trust in the host (may require reimaging).
- **Credential replay / spraying** bypass weak auth — MFA is the key defense.
- **Missing logs** cripple investigation — protect log integrity.
- **DDoS** denies availability (the A in CIA).

---

## 14. Attacker / Adversary Perspective

- Attackers **suppress indicators** (clear logs, use rootkits, low-and-slow spraying).
- They prefer **credential reuse/replay** over noisy brute force when possible.
- **Amplification/reflection** lets them generate huge volume cheaply.
- **Downgrade** attacks exploit backward-compatibility.
- **Living-off-the-land** avoids malware signatures entirely.

---

## 15. Defender Perspective

- **Centralize and protect logs** (so "missing logs" itself is detectable).
- **MFA** to defeat spraying/brute force/replay.
- **Rate-limiting + lockout + CAPTCHA** for password attacks.
- **EDR + behavior analytics** for malware and stealth.
- **DDoS protection / anti-amplification** (disable open resolvers, ingress filtering).
- **UEBA** for impossible travel and concurrent sessions.

---

## 16. Detection Techniques

- **SIEM correlation** of failed logins, lockouts, out-of-cycle/missing logs.
- **UEBA** for impossible travel, concurrent sessions, anomalous behavior.
- **EDR / anti-malware / behavior detection** for the 9 malware types.
- **NetFlow / IDS** for DDoS, on-path, DNS attacks.
- **WAF logs** for injection, traversal, forgery.
- **File-integrity & log-integrity monitoring** (missing logs, rootkits).

---

## 17. Mitigation Techniques

- **MFA, lockout, rate-limiting** (password/credential attacks).
- **Patching + EDR + least privilege** (malware, privilege escalation).
- **Network segmentation** (worm containment).
- **DDoS mitigation services + anti-amplification** config.
- **TLS hardening / disable weak protocols** (downgrade).
- **Strong hashes (SHA-256+) with salt** (collision/birthday).
- **Input validation** (injection, traversal, forgery).
- **Backups** (ransomware recovery).

---

## 18. Best Practices

1. **Centralize, protect, and monitor logs** (detect missing/out-of-cycle).
2. **Deploy MFA everywhere** — it defeats most credential attacks.
3. **Use EDR + behavior detection**, not just signatures.
4. **Segment networks** to contain worms.
5. **Harden TLS / disable weak ciphers** (downgrade/crypto attacks).
6. **Rate-limit and lock out** repeated auth failures.
7. **Maintain tested backups** for ransomware.
8. **Tune UEBA** for impossible travel / concurrent sessions.

## 20. Diagrams

**Virus vs. worm:**

```
   VIRUS:  needs a host file + a USER to run it ──► then spreads
   WORM:   self-replicates across the NETWORK ──► no user needed
```

**Amplified vs. reflected DDoS:**

```
   AMPLIFIED:  small request ─► open service ─► HUGE response ─► victim
   REFLECTED:  attacker spoofs victim's IP ─► 3rd parties reply ─► victim
```

**Spraying vs. brute force:**

```
   BRUTE FORCE:  1 account  ← many passwords   (loud, triggers lockout)
   SPRAYING:     many accounts ← few passwords  (quiet, evades lockout)
```

## 21. Exam Tips

- **Virus needs a user to execute it; worm self-spreads; trojan is disguised.** (The top malware trap.)
- **Rootkit = stealth/kernel-level hiding.** **Logic bomb = triggers on a condition.**
- **Spraying = few passwords across many accounts (evades lockout); brute force = many passwords on one account.**
- **Impossible travel = stolen credentials.**
- **Missing logs = attacker hiding tracks.**
- **Resource inaccessibility = ransomware/DoS; resource consumption = malware/mining/DDoS.**
- **On-path = old "man-in-the-middle."**
- **Amplified = big response from small request; reflected = bounced off third parties (spoofed source).**
- **Directory traversal = `../`** to escape the web root.
- **Downgrade = force weaker crypto; collision = same hash; birthday = the collision-finding technique.**
- **Replay/credential replay = reuse captured credentials/requests** (MFA + nonces defeat it).

## 29. Practice Questions

**Q1.** A piece of malware copies itself and spreads across the network without any user interaction. What type is it?
A. Virus  B. Trojan  C. Worm  D. Rootkit

**Q2.** An attacker tries the single password "Password123" against 500 different user accounts to avoid triggering lockouts. This is:
A. Brute force  B. Password spraying  C. Credential replay  D. Birthday attack

**Q3.** A user's account shows successful logins from London and Sydney just three minutes apart. Which indicator is this?
A. Account lockout  B. Impossible travel  C. Out-of-cycle logging  D. Resource consumption

**Q4.** Malware that disguises itself as a legitimate program to trick users into installing it is a:
A. Worm  B. Trojan  C. Logic bomb  D. Virus

**Q5.** Security logs for a critical two-hour window are completely absent after an incident. This indicator suggests:
A. Resource consumption  B. Impossible travel  C. Missing logs (anti-forensics)  D. Concurrent sessions

**Q6.** An attacker forces a TLS connection to fall back to an older, weaker protocol version to exploit it. This is a:
A. Collision attack  B. Downgrade attack  C. Birthday attack  D. Replay attack

**Q7.** A web request contains `../../../etc/passwd`. Which attack is this?
A. SQL injection  B. Directory traversal  C. Privilege escalation  D. Forgery

**Q8.** Two different documents are found to produce the exact same hash value. This is a:
A. Downgrade  B. Collision  C. Buffer overflow  D. Replay

**Q9.** Malware hides itself at the kernel level so standard tools cannot detect it. This is a:
A. Spyware  B. Keylogger  C. Rootkit  D. Bloatware

**Q10.** In an amplified DDoS attack, the attacker:
A. Sends large requests directly to the victim
B. Sends small requests to services that return large responses to the victim
C. Tries many passwords against one account
D. Encrypts the victim's files

### Answer Key & Explanations

- **Q1 — C (Worm).** Self-replicates, spreads without user action.
- **Q2 — B (Password spraying).** Few passwords across many accounts to evade lockout.
- **Q3 — B (Impossible travel).** Distant logins too close in time = stolen creds.
- **Q4 — B (Trojan).** Defined by disguise.
- **Q5 — C (Missing logs).** Gaps suggest the attacker cleared logs.
- **Q6 — B (Downgrade).** Forcing a weaker protocol/cipher.
- **Q7 — B (Directory traversal).** `../` escapes the web root.
- **Q8 — B (Collision).** Two inputs, same hash.
- **Q9 — C (Rootkit).** Kernel-level stealth.
- **Q10 — B.** Amplification: small request → large response at the victim.

## 30. PBQ-style Scenario

> **Scenario:** Your SIEM surfaces several events. Identify the attack or malware type, and the key indicator.

| # | Observation | Attack / type | Indicator |
|---|-------------|---------------|-----------|
| 1 | Files across a share are encrypted; a ransom note appears; data is unavailable | ? | ? |
| 2 | One account: 2,000 failed logins in 10 minutes, then a lockout | ? | ? |
| 3 | The password "Summer2024!" is attempted once each against 800 accounts | ? | ? |
| 4 | A login from two continents within minutes | ? | ? |
| 5 | A host's CPU pegs at 100% from an unknown process mining crypto | ? | ? |
| 6 | Event logs for the breach window are gone | ? | ? |
| 7 | A program self-propagates to every machine on the LAN unaided | ? | ? |
| 8 | A server is tricked into using SSL 3.0 instead of TLS 1.3 | ? | ? |

**Answer Key:**

1. **Ransomware** — indicator: **resource inaccessibility**
2. **Brute force** (password) — indicator: **account lockout**
3. **Password spraying** — indicator: many accounts, few passwords (low-and-slow auth failures)
4. **Stolen credentials / account takeover** — indicator: **impossible travel**
5. **Malware (cryptominer)** — indicator: **resource consumption**
6. **Anti-forensics** — indicator: **missing logs**
7. **Worm** — indicator: rapid self-propagation across hosts
8. **Downgrade attack** (cryptographic)
